## 模式定位

单例模式的目标很简单：让一个类在当前 JVM 进程内只创建一个对象，并提供一个统一入口获取它。

典型代码使用方式大概是这样：

```java
ConfigManager configManager = ConfigManager.getInstance();
String value = configManager.get("app.name");
```

调用方不需要关心对象什么时候创建，也不应该随便 `new` 一个新的实例。

这种模式适合一些“全局只需要一份”的对象：

- 配置管理器；
- 本地缓存容器；
- ID 生成器；
- 线程池管理器；
- 应用级别上下文；
- 某些工具类的状态管理对象。

但单例不是万能的。它能减少重复创建，也能统一访问入口；同时也可能带来全局状态、测试困难、隐藏依赖等问题。

## 饿汉式

饿汉式在类加载时就创建实例。

```java
public class ConfigManager {

    // 类加载时直接初始化，JVM 保证类初始化过程是线程安全的
    private static final ConfigManager INSTANCE = new ConfigManager();

    // 私有构造方法，避免外部直接 new 对象
    private ConfigManager() {
    }

    public static ConfigManager getInstance() {
        return INSTANCE;
    }

    public String get(String key) {
        // 示例代码：真实项目里通常会读取配置文件或配置中心
        return System.getProperty(key);
    }
}
```

优点很明显：

- 写法简单；
- 没有并发问题；
- 获取实例时不需要加锁。

缺点也很明显：

- 类一加载就创建对象；
- 如果对象初始化成本高，可能造成启动变慢；
- 如果这个单例一直没被使用，也会浪费资源。

如果对象很轻量，并且应用启动后基本一定会用到，饿汉式可以接受。

## 懒汉式

懒汉式的想法是：第一次用到时再创建。

很多人刚开始会写成这样：

```java
public class IdGenerator {

    private static IdGenerator instance;

    private IdGenerator() {
    }

    public static IdGenerator getInstance() {
        if (instance == null) {
            // 多线程同时进入这里时，可能创建多个对象
            instance = new IdGenerator();
        }
        return instance;
    }
}
```

这段代码在单线程下没问题，多线程下就不安全。

假设两个线程同时调用 `getInstance()`：

```text
线程 A 判断 instance == null
线程 B 判断 instance == null
线程 A 创建对象
线程 B 也创建对象
```

这样就违背了“全局唯一实例”的要求。

## 同步方法

可以直接给方法加锁：

```java
public class IdGenerator {

    private static IdGenerator instance;

    private IdGenerator() {
    }

    public static synchronized IdGenerator getInstance() {
        if (instance == null) {
            // 加锁后同一时间只有一个线程能执行创建逻辑
            instance = new IdGenerator();
        }
        return instance;
    }
}
```

这种写法是线程安全的，逻辑也容易理解。

问题是每次获取实例都要进入同步方法。对象创建完成后，后续调用只是返回实例，却仍然要竞争锁。

在低频调用场景，这个问题不严重；在高频调用场景，就不太合适。

## 双重检查锁

双重检查锁，也就是 Double-Checked Locking，常见写法如下：

```java
public class IdGenerator {

    // volatile 用来禁止指令重排序，保证其他线程读到的是初始化完成的对象
    private static volatile IdGenerator instance;

    private IdGenerator() {
    }

    public static IdGenerator getInstance() {
        if (instance == null) {
            // 第一次检查为空时才进入同步块，避免每次调用都加锁
            synchronized (IdGenerator.class) {
                if (instance == null) {
                    // 第二次检查是为了防止多个线程排队进入后重复创建
                    instance = new IdGenerator();
                }
            }
        }
        return instance;
    }
}
```

这里有两个关键点。

第一个关键点是两次 `if (instance == null)`。

第一次判断是为了减少锁竞争；第二次判断是为了保证已经有线程创建实例后，后续线程不会重复创建。

第二个关键点是 `volatile`。

对象创建并不是一个完全不可拆分的动作，可以粗略理解成：

```text
1. 分配内存
2. 初始化对象
3. 引用指向内存地址
```

如果没有 `volatile`，在某些情况下可能发生指令重排序，其他线程可能拿到一个“还没初始化完成”的对象引用。

所以双重检查锁必须配合 `volatile` 使用。

## 静态内部类

静态内部类写法兼顾了懒加载和线程安全。

```java
public class ConfigManager {

    private ConfigManager() {
    }

    public static ConfigManager getInstance() {
        return Holder.INSTANCE;
    }

    private static class Holder {
        // Holder 类被主动使用时才会初始化 INSTANCE
        private static final ConfigManager INSTANCE = new ConfigManager();
    }

    public String get(String key) {
        return System.getProperty(key);
    }
}
```

这种写法依赖 JVM 的类加载机制：

- 外部类 `ConfigManager` 加载时，不会立刻初始化 `Holder`；
- 第一次调用 `getInstance()` 时，才会触发 `Holder` 初始化；
- 类初始化过程由 JVM 保证线程安全。

它的优点是：

```text
懒加载
线程安全
代码不复杂
获取实例时没有显式锁竞争
```

很多普通 Java 单例场景，静态内部类是比较稳妥的选择。

## 枚举单例

Effective Java 里推荐过枚举单例写法。

```java
public enum GlobalIdGenerator {

    INSTANCE;

    public long nextId() {
        // 示例代码：真实项目里要考虑机器号、时钟回拨、序列号等问题
        return System.currentTimeMillis();
    }
}
```

使用方式：

```java
long id = GlobalIdGenerator.INSTANCE.nextId();
```

枚举单例的优势：

- 写法非常简洁；
- 天然线程安全；
- 可以避免反射破坏普通私有构造方法；
- 序列化和反序列化时也能保证单例语义。

它的缺点是写法不如普通类灵活。比如需要继承某个类时，枚举就不适合。

如果只是做一个稳定的全局唯一对象，枚举单例是很强的选择。

## Spring Singleton

Spring 里默认 Bean 作用域是 singleton，例如：

```java
@Service
public class OrderService {
}
```

大多数情况下，Spring 容器里只会创建一个 `OrderService` Bean。

但 Spring singleton 和设计模式里的单例不是完全一回事。

设计模式里的单例通常强调：

```text
类自己控制实例创建
外部通过静态方法获取唯一实例
构造方法通常私有化
```

Spring singleton 更强调：

```text
对象由 Spring 容器创建和管理
在同一个 Spring 容器内只有一个 Bean 实例
通过依赖注入获取对象
```

也就是说，Spring 的单例是“容器级别单例”，不是“JVM 全局绝对单例”。

如果一个应用里启动多个 Spring 容器，同一个 Bean 可能会有多个实例。

## 常见误区

### 可变状态

单例对象如果保存可变状态，要特别小心。

下面这种写法就很危险：

```java
public class LoginContext {

    private static final LoginContext INSTANCE = new LoginContext();

    // 这个字段会被所有线程共享，容易发生用户信息串号
    private Long currentUserId;

    private LoginContext() {
    }

    public static LoginContext getInstance() {
        return INSTANCE;
    }

    public void setCurrentUserId(Long currentUserId) {
        this.currentUserId = currentUserId;
    }

    public Long getCurrentUserId() {
        return currentUserId;
    }
}
```

Web 项目里多个请求线程同时访问这个对象，`currentUserId` 就会变成全局共享变量。

用户 A 刚设置完，用户 B 可能立刻覆盖它，最终出现数据错乱。

更合适的方式是使用方法参数、局部变量、`ThreadLocal`，或者由框架提供请求级上下文。

### 隐藏依赖

有些代码把所有工具都写成单例，然后到处 `XxxManager.getInstance()`。

短期看起来少写了依赖注入，长期会带来几个问题：

- 类之间依赖关系不清楚；
- 单元测试难替换依赖；
- 初始化顺序不好控制；
- 全局状态越来越多；
- 代码和具体实现强耦合。

在 Spring Boot 项目里，普通业务对象优先交给容器管理，不需要手写大量单例。

### 序列化问题

普通单例类如果实现了 `Serializable`，反序列化时可能创建新对象。

可以通过 `readResolve()` 修复：

```java
public class ConfigManager implements Serializable {

    private static final long serialVersionUID = 1L;

    private static final ConfigManager INSTANCE = new ConfigManager();

    private ConfigManager() {
    }

    public static ConfigManager getInstance() {
        return INSTANCE;
    }

    private Object readResolve() {
        // 反序列化时返回已有实例，避免生成新的对象
        return INSTANCE;
    }
}
```

如果确实需要序列化，又想天然避免这个问题，可以考虑枚举单例。

### 反射问题

普通私有构造方法并不是绝对安全的，反射可以强行调用。

```java
Constructor<ConfigManager> constructor = ConfigManager.class.getDeclaredConstructor();
constructor.setAccessible(true);
ConfigManager newInstance = constructor.newInstance();
```

这也是枚举单例更安全的原因之一。

当然，普通业务系统通常不需要把这个问题复杂化。了解风险即可，真正的重点还是避免在项目里随意制造全局状态。

## 选型建议

可以简单按下面的方式判断：

| 场景 | 建议 |
| --- | --- |
| Spring Boot 普通业务服务 | 交给 Spring 容器管理 |
| 轻量对象，启动后一定使用 | 饿汉式 |
| 需要懒加载，普通 Java 类 | 静态内部类 |
| 极简全局唯一工具对象 | 枚举单例 |
| 高频调用且需要懒加载 | 静态内部类或 DCL |
| 保存用户请求状态 | 不要做成单例 |

如果没有特殊原因，不建议优先使用双重检查锁。它虽然常见，但细节更多，写错后问题不容易暴露。

## 业务示例

假设项目里需要一个本地敏感词词典，启动后不一定立刻使用，但第一次审核内容时要加载词库。

可以用静态内部类实现：

```java
public class SensitiveWordHolder {

    private final Set<String> words;

    private SensitiveWordHolder() {
        // 构造方法里加载词库，避免每次校验都读取文件
        this.words = loadWords();
    }

    public static SensitiveWordHolder getInstance() {
        return Holder.INSTANCE;
    }

    public boolean contains(String content) {
        if (content == null || content.isBlank()) {
            return false;
        }

        // 示例实现：真实项目里可以用 DFA、AC 自动机等方式优化匹配性能
        for (String word : words) {
            if (content.contains(word)) {
                return true;
            }
        }
        return false;
    }

    private Set<String> loadWords() {
        Set<String> result = new HashSet<>();

        // 示例代码：真实项目里可以读取 classpath 文件、数据库或配置中心
        result.add("测试敏感词");
        result.add("违规内容");
        return Collections.unmodifiableSet(result);
    }

    private static class Holder {
        private static final SensitiveWordHolder INSTANCE = new SensitiveWordHolder();
    }
}
```

使用时：

```java
boolean hit = SensitiveWordHolder.getInstance().contains("这是一段需要审核的内容");
```

这个例子里，词库加载后不希望被随意修改，所以返回了不可变集合。单例对象里如果保存共享数据，最好让数据尽量不可变，这样并发风险会低很多。

## 总结

单例模式的价值在于控制对象创建，保证全局访问入口统一。

Java 里常见写法包括饿汉式、同步懒汉式、双重检查锁、静态内部类、枚举单例。普通业务开发里，静态内部类和枚举单例更推荐；Spring Boot 项目里的业务对象，优先交给 Spring 容器管理。

真正需要注意的是：不要把单例当成全局变量收纳箱。只要单例对象里保存了可变状态，就要认真考虑线程安全、测试替换和生命周期管理问题。
