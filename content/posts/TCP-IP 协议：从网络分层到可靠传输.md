## 写在前面

平时写业务代码时，我们更多接触的是 HTTP、RPC、WebSocket、数据库连接、Redis 连接这些上层概念。它们看起来是不同的技术，但底层都离不开网络通信。

TCP/IP 就是互联网通信的基础协议族。无论是浏览器访问网页，还是服务之间互相调用，本质上都需要把数据从一台机器传到另一台机器。

如果说 HTTP 关注的是应用层请求和响应，那么 TCP/IP 更关注数据在网络中如何传输：

```text
数据怎么封装
IP 怎么寻址
路由怎么转发
TCP 怎么建立连接
数据如何可靠传输
连接如何关闭
```

这篇文章从网络分层开始，梳理 TCP/IP 协议中最核心、最常用的知识点。

## TCP/IP 是什么

TCP/IP 不是单指 TCP 和 IP 两个协议，而是一组网络协议族。

常见协议包括：

| 协议 | 作用 |
| --- | --- |
| IP | 负责寻址和路由，把数据包从源主机送到目标主机 |
| TCP | 面向连接，提供可靠传输 |
| UDP | 无连接，传输开销小，不保证可靠 |
| ICMP | 网络控制和诊断，例如 ping |
| ARP | 根据 IP 地址找到局域网内对应的 MAC 地址 |
| DNS | 把域名解析成 IP 地址 |
| HTTP | 应用层协议，基于 TCP 或 QUIC 等传输协议 |

可以简单理解为：

```text
IP 解决“发到哪台机器”
TCP 解决“如何可靠地发过去”
UDP 解决“尽快发过去，不保证一定到”
HTTP 解决“客户端和服务端传什么内容”
```

## 网络分层模型

学习 TCP/IP，先要理解网络分层。

常见有两种模型：OSI 七层模型和 TCP/IP 四层模型。

| OSI 七层模型 | TCP/IP 四层模型 | 常见协议或设备 |
| --- | --- | --- |
| 应用层 | 应用层 | HTTP、DNS、FTP、SMTP、WebSocket |
| 表示层 | 应用层 | 编码、压缩、加密等 |
| 会话层 | 应用层 | 会话管理 |
| 传输层 | 传输层 | TCP、UDP |
| 网络层 | 网络层 | IP、ICMP、路由器 |
| 数据链路层 | 网络接口层 | Ethernet、ARP、交换机 |
| 物理层 | 网络接口层 | 网线、光纤、无线信号 |

理解网络分层时，不需要机械背模型，但要知道每层大概负责什么：

```text
应用层：传什么业务数据，例如 HTTP 请求、MySQL 协议、Redis 协议
传输层：进程到进程，例如 TCP 端口、连接、重传、拥塞控制
网络层：主机到主机，例如 IP 地址、路由、ICMP
链路层：局域网内传输，例如 MAC 地址、交换机、ARP
物理层：真实介质，例如网线、网卡、无线信号
```

分层的好处是职责清晰。HTTP 不需要关心网线怎么传输，TCP 不需要关心 JSON 业务字段，IP 不需要关心应用层返回的是 HTML 还是图片。

## 数据是怎么封装的

假设浏览器发送一个 HTTP 请求，数据会从应用层一路向下封装：

```text
HTTP 数据
  ↓ 加 TCP 头
TCP Segment
  ↓ 加 IP 头
IP Packet
  ↓ 加以太网帧头帧尾
Ethernet Frame
  ↓ 通过网卡发送
```

接收方则反过来拆包：

```text
Ethernet Frame
  ↓ 去掉以太网头
IP Packet
  ↓ 去掉 IP 头
TCP Segment
  ↓ 去掉 TCP 头
HTTP 数据
```

可以理解为快递打包：

```text
业务数据：真正要寄的东西
TCP 头：收件进程、发件进程、序号、确认号
IP 头：源 IP、目标 IP
以太网头：源 MAC、目标 MAC
```

每一层只处理自己关心的信息，然后把剩下的数据交给上一层或下一层。

## IP：寻址和路由

IP 协议负责把数据包从源主机送到目标主机。

一个 IPv4 地址示例：

```text
192.168.1.10
```

IPv4 地址是 32 位，通常写成四段十进制数字。IPv6 地址更长，用来解决 IPv4 地址空间不够的问题。

### IP 地址和端口不是一回事

很多刚接触网络问题时容易混淆 IP 和端口。

```text
IP 地址：定位一台机器
端口：定位机器上的一个进程或服务
```

比如：

```text
192.168.1.10:8080
```

含义是访问 `192.168.1.10` 这台机器上的 `8080` 端口。

同一台机器上可以同时运行多个服务：

| 服务 | 地址 |
| --- | --- |
| Nginx | `192.168.1.10:80` |
| Spring Boot | `192.168.1.10:8080` |
| MySQL | `192.168.1.10:3306` |
| Redis | `192.168.1.10:6379` |

IP 定位主机，端口定位进程。

### 子网掩码

子网掩码用来判断两个 IP 是否在同一个网段。

例如：

```text
IP:      192.168.1.10
Mask:    255.255.255.0
Gateway: 192.168.1.1
```

`255.255.255.0` 表示前三段是网络号，最后一段是主机号。

所以：

```text
192.168.1.10 和 192.168.1.20 在同一网段
192.168.1.10 和 192.168.2.20 不在同一网段
```

如果目标 IP 在同一网段，可以直接通过二层网络发送；如果不在同一网段，就要把数据交给默认网关。

### 默认网关

默认网关可以理解为当前网络的出口。

当一台机器不知道怎么直达目标 IP 时，就把数据包交给网关：

```text
本机 -> 默认网关 -> 路由器 -> 目标网络 -> 目标机器
```

很多“服务器能访问内网，不能访问外网”的问题，都和网关、路由、防火墙、NAT 配置有关。

### 路由转发

路由器根据路由表决定下一跳。

可以用下面命令查看本机路由：

```bash
ip route
```

常见输出类似：

```text
default via 192.168.1.1 dev eth0
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.10
```

含义大致是：

```text
访问 192.168.1.0/24 网段，直接从 eth0 发出去
其他目标地址，走默认网关 192.168.1.1
```

如果路由配置错误，可能会出现：

```text
No route to host
Network is unreachable
```

## ARP：通过 IP 找 MAC

在局域网内真正发送数据帧时，需要知道目标机器的 MAC 地址。

ARP 的作用是：根据 IP 地址查询对应的 MAC 地址。

流程大致是：

```text
主机 A 想访问 192.168.1.20
-> A 先查自己的 ARP 缓存
-> 如果没有，就广播：谁是 192.168.1.20？
-> 目标主机回复自己的 MAC 地址
-> A 记录到 ARP 缓存
-> 后续直接发送以太网帧
```

查看 ARP 缓存：

```bash
arp -a
# 或
ip neigh
```

ARP 通常只在局域网内工作，跨网段访问时，本机会先找到网关的 MAC 地址，然后把数据发给网关。

## ICMP：网络诊断

ICMP 常用于网络控制和诊断。最常见的工具就是 `ping`。

```bash
ping www.baidu.com
```

如果能 ping 通，说明至少有一部分网络链路是通的。但 ping 不通不一定代表服务不可用，因为很多服务器或防火墙会禁用 ICMP。

比如：

```text
ping 不通，但 curl 能访问 HTTP 服务
```

这种情况并不少见。

所以要区分：

```text
ping 检查的是 ICMP
curl 检查的是应用层 HTTP
nc/telnet 检查的是 TCP 端口连通性
```

## TCP 和 UDP 的区别

传输层最常见的是 TCP 和 UDP。

| 对比项 | TCP | UDP |
| --- | --- | --- |
| 是否连接 | 面向连接 | 无连接 |
| 可靠性 | 保证可靠传输 | 不保证可靠传输 |
| 顺序性 | 保证有序 | 不保证有序 |
| 开销 | 较高 | 较低 |
| 速度 | 相对慢 | 相对快 |
| 典型场景 | HTTP、HTTPS、MySQL、Redis、SSH | DNS、直播、音视频、游戏、日志上报 |

TCP 更像打电话：先建立连接，再持续通信。

UDP 更像发短信：直接发出去，不保证对方一定收到，也不保证顺序。

这并不代表 UDP 不可靠就没用。很多实时场景宁可丢一部分数据，也不希望因为重传导致延迟升高。

## TCP 连接由什么标识

一个 TCP 连接由四元组唯一标识：

```text
源 IP
源端口
目标 IP
目标端口
```

例如：

```text
客户端 10.0.0.5:52341 -> 服务端 10.0.0.10:8080
```

这里 `52341` 是客户端临时端口，`8080` 是服务端监听端口。

同一个客户端可以和同一个服务端建立多个 TCP 连接，只要源端口不同即可。

查看连接：

```bash
ss -antp
```

常见输出里会看到：

```text
ESTAB      0      0      10.0.0.5:52341     10.0.0.10:8080
LISTEN     0      128    0.0.0.0:8080       0.0.0.0:*
```

`LISTEN` 表示服务正在监听端口，`ESTAB` 表示连接已经建立。

## TCP 三次握手

TCP 是面向连接的协议。通信前需要先建立连接，这个过程就是三次握手。

```text
Client                              Server
  |                                   |
  | -------- SYN seq=x ------------> |
  |                                   |
  | <--- SYN + ACK seq=y ack=x+1 ---- |
  |                                   |
  | -------- ACK ack=y+1 ----------> |
  |                                   |
  |        Connection Established     |
```

三次握手做了几件事：

1. 客户端确认自己能发送；
2. 服务端确认自己能接收，也能发送；
3. 客户端确认自己能接收；
4. 双方同步初始序列号。

### 为什么不是两次握手

两次握手不能让服务端确认客户端的接收能力。

如果只有两次：

```text
Client -> SYN -> Server
Client <- SYN+ACK <- Server
```

服务端发完 `SYN+ACK` 后就认为连接建立了，但客户端是否收到并确认，服务端并不知道。

三次握手通过最后一次 ACK，让服务端知道客户端已经收到自己的响应，连接状态更可靠。

### SYN 队列和全连接队列

服务端处理 TCP 建连时，通常涉及两个队列：

```text
半连接队列：收到 SYN，还没完成三次握手
全连接队列：三次握手完成，等待应用 accept
```

如果队列满了，可能出现连接失败、连接超时、请求进不来等问题。

高并发场景下，需要关注：

- 服务端 backlog 配置；
- 系统内核参数；
- 应用 accept 是否及时；
- 是否存在 SYN Flood 攻击；
- 负载均衡是否健康。

## TCP 四次挥手

TCP 连接关闭通常需要四次挥手。

```text
Client                              Server
  |                                   |
  | -------- FIN -------------------> |
  |                                   |
  | <------- ACK -------------------- |
  |                                   |
  |                  Server 处理剩余数据
  |                                   |
  | <------- FIN -------------------- |
  |                                   |
  | -------- ACK -------------------> |
  |                                   |
  |              Closed               |
```

为什么关闭连接要四次？

因为 TCP 是全双工的。客户端不想发数据了，不代表服务端也没有数据要发。

所以关闭过程可以理解为：

```text
客户端：我这边发完了
服务端：知道了
服务端：我这边也发完了
客户端：知道了
```

## TIME_WAIT 和 CLOSE_WAIT

观察 TCP 连接状态时，经常会看到 `TIME_WAIT` 和 `CLOSE_WAIT`。

### TIME_WAIT

主动关闭连接的一方，通常会进入 `TIME_WAIT` 状态。

它的作用主要有两个：

1. 确保最后一个 ACK 能被对方收到；
2. 避免旧连接中的延迟报文影响新连接。

`TIME_WAIT` 多不一定是问题。短连接很多的服务，本来就可能产生大量 `TIME_WAIT`。

需要关注的是：

- 是否存在大量短连接；
- 是否没有开启连接复用；
- 客户端本地端口是否耗尽；
- 连接池配置是否合理。

### CLOSE_WAIT

`CLOSE_WAIT` 通常表示对方已经关闭连接，但本地应用还没有关闭 socket。

如果 `CLOSE_WAIT` 很多，往往说明应用层没有正确释放连接，例如：

- HTTP 响应体没有关闭；
- 数据库连接没有归还；
- Socket 使用后没有 close；
- 异常分支漏掉资源释放。

相比 `TIME_WAIT`，大量 `CLOSE_WAIT` 更值得警惕。

查看连接状态统计：

```bash
ss -ant | awk 'NR>1 {count[$1]++} END {for (s in count) print s, count[s]}'
```

## TCP 如何保证可靠传输

IP 层只负责尽力而为地发送数据包，不保证一定到达，也不保证顺序。

TCP 在 IP 之上提供可靠传输，主要依靠这些机制：

```text
序列号
确认应答
超时重传
滑动窗口
流量控制
拥塞控制
校验和
```

### 序列号和确认号

TCP 会给发送的字节流编号。

假设客户端发送了 1000 字节，序列号从 100 开始：

```text
发送：seq=100，len=1000
接收方确认：ack=1100
```

`ack=1100` 表示：1100 之前的数据我都收到了，下一次希望从 1100 开始发。

### 超时重传

如果发送方发出数据后，迟迟没有收到 ACK，就会认为数据可能丢了，然后重传。

```text
发送数据
-> 等待 ACK
-> 超时未收到
-> 重传数据
```

但超时时间不能太短，否则会导致大量不必要重传；也不能太长，否则丢包恢复太慢。TCP 会根据网络 RTT 动态估算重传超时时间。

### 滑动窗口

如果每发一小段数据都要等一次 ACK，效率会很低。

滑动窗口允许发送方在未收到 ACK 前连续发送多段数据：

```text
可发送窗口：[已经发送未确认的数据 + 还能继续发送的数据]
```

ACK 不断返回后，窗口向前滑动，发送方可以继续发送新数据。

### 流量控制

流量控制是为了避免发送方发得太快，把接收方缓冲区打满。

接收方会通过 TCP 头里的窗口大小告诉发送方：

```text
我现在还能接收多少数据
```

如果接收方处理不过来，窗口会变小；如果缓冲区满了，可能出现零窗口。

### 拥塞控制

流量控制关注接收方能不能接住，拥塞控制关注网络链路能不能承受。

拥塞控制常见机制包括：

- 慢启动；
- 拥塞避免；
- 快速重传；
- 快速恢复。

可以简单理解为：

```text
刚开始不要发太猛
网络状态好就逐渐加速
发现丢包或拥塞就降速
```

这也是为什么同一个接口在不同网络环境下耗时差异很大。应用代码没变，网络拥塞、丢包、RTT 变化都会影响整体耗时。

## 粘包和半包

TCP 是字节流协议，不保留应用层消息边界。

这句话很重要。

假设应用层发送两条消息：

```text
hello
world
```

接收方看到的可能是：

```text
helloworld
```

这就是粘包。

也可能看到：

```text
hel
loworld
```

这就是半包。

原因是 TCP 只保证字节有序可靠，不知道业务消息应该从哪里开始、在哪里结束。

解决方式要在应用层协议里定义边界，常见方案：

| 方案 | 说明 |
| --- | --- |
| 固定长度 | 每条消息固定大小，不足补齐 |
| 分隔符 | 用 `\n` 等特殊字符分隔消息 |
| 长度字段 | 消息头里记录 body 长度 |
| TLV | Type、Length、Value 格式 |

HTTP 使用 Header 和 Body 长度、Chunked 等机制解决消息边界问题。自定义 TCP 协议时一定要设计清楚拆包规则。

## MTU 和 MSS

MTU 是链路层最大传输单元，常见以太网 MTU 是 1500 字节。

MSS 是 TCP 单个报文段能承载的最大应用数据长度。

关系大致是：

```text
MSS = MTU - IP 头 - TCP 头
```

IPv4 下，如果 MTU 是 1500，IP 头 20 字节，TCP 头 20 字节，那么 MSS 通常是：

```text
1500 - 20 - 20 = 1460
```

如果数据包超过链路 MTU，可能会发生分片，或者因为禁止分片导致发送失败。

一些 VPN、容器网络、云网络环境中，MTU 配置不一致可能导致很隐蔽的问题：

```text
小包正常
大包失败
偶发超时
TLS 握手卡住
接口上传文件失败
```

这类现象往往和 MTU 配置有关。

## NAT：内网访问外网的关键

很多机器使用的是内网 IP，例如：

```text
10.0.0.5
192.168.1.10
172.16.0.20
```

这些地址不能直接在公网路由。内网机器访问外网时，通常需要 NAT。

NAT 会把内网地址转换成公网地址：

```text
内网机器 192.168.1.10:52341
-> NAT 网关 公网IP:40001
-> 目标服务器 8.8.8.8:53
```

响应回来时，NAT 网关再根据映射表转回内网机器。

NAT 相关问题常见于：

- 容器访问外网；
- 云服务器访问公网接口；
- 多个服务共享出口 IP；
- 第三方平台需要配置 IP 白名单；
- NAT 连接数耗尽导致外部接口偶发失败。

如果第三方说“请求来源 IP 不在白名单”，要确认的通常不是应用服务器内网 IP，而是实际出口公网 IP。

## DNS：域名解析

应用里访问的通常是域名：

```text
api.example.com
```

真正建立连接前，需要先把域名解析成 IP。

DNS 解析大致过程：

```text
浏览器/应用缓存
-> 操作系统缓存
-> 本地 DNS 服务器
-> 根域名服务器
-> 顶级域名服务器
-> 权威 DNS 服务器
-> 返回 IP
```

实际会有多级缓存，不是每次都完整走一遍。

常用命令：

```bash
nslookup api.example.com
```

```bash
dig api.example.com
```

DNS 需要注意的点：

- 域名解析到旧 IP；
- 本地 DNS 缓存未刷新；
- 不同地区解析结果不同；
- 内网 DNS 和公网 DNS 结果不同；
- 容器内 DNS 配置错误；
- 域名 TTL 设置过长，切流量不及时。

所以域名解析并不只是把域名变成 IP，还会受到缓存、TTL、地域和内外网环境的影响。

## 总结

TCP/IP 是互联网通信的基础。日常开发中即使不直接操作 socket，也会不断和它打交道。

可以按这条主线理解：

```text
应用层产生数据
-> TCP/UDP 负责进程间传输
-> IP 负责主机间寻址和路由
-> 链路层负责局域网内传输
-> 物理层负责真实信号传输
```

几个核心点：

- IP 负责寻址和路由；
- 端口用于定位主机上的进程；
- TCP 通过三次握手建立连接；
- TCP 通过四次挥手关闭连接；
- TCP 通过序列号、ACK、重传、滑动窗口保证可靠传输；
- 流量控制保护接收方，拥塞控制保护网络链路；
- TCP 是字节流协议，应用层需要自己处理消息边界；
- DNS、NAT、路由等机制共同支撑一次网络请求；
- `TIME_WAIT` 不一定是问题，大量 `CLOSE_WAIT` 更值得警惕；

理解 TCP/IP，不是为了记住所有协议细节，而是为了建立清晰的网络通信模型：应用数据如何被封装，如何找到目标主机，如何建立连接，又如何保证数据可靠传输。
