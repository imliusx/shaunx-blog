## Agent 不是一个新模型

AI Agent 经常被翻译成“智能体”。刚接触这个概念时，很容易把它理解成某个新的大模型，或者某种特殊算法。

更准确的理解是：Agent 是一种应用形态。

大模型负责理解语言、生成计划、判断下一步动作；工具负责执行真实操作；业务系统负责提供数据和能力；Agent 框架把这些部分串起来，让一次对话可以变成一段可执行的任务流程。

普通聊天机器人更像是“回答问题”：

```text
用户提问
-> 大模型生成回答
-> 对话结束
```

Agent 更像是“带着目标做事”：

```text
用户提出目标
-> Agent 理解目标
-> 拆分任务步骤
-> 判断需要哪些工具
-> 调用工具获取结果
-> 根据结果决定继续执行还是结束
-> 输出最终结果
```

区别不在于回答是否自然，而在于能不能持续推进任务。

## 一个后端开发更容易理解的例子

假设公司内部有一个 AI 运维助手，用户输入：

```text
帮我看一下订单服务最近 30 分钟为什么大量超时。
```

普通大模型只能给出一些排查建议，比如检查 CPU、数据库、网络、日志等。

Agent 可以接入真实工具后执行更完整的流程：

```text
1. 查询订单服务最近 30 分钟的接口耗时指标
2. 查询服务实例 CPU、内存、GC 情况
3. 检索 error 日志和慢 SQL 日志
4. 对比发布时间和报警时间
5. 汇总可能原因
6. 给出下一步处理建议
```

如果工具设计得足够完善，它甚至可以继续执行：

```text
- 创建故障记录
- 通知值班群
- 回滚最近一次发布
- 拉取相关日志文件
- 生成排查报告
```

这些动作背后不是大模型“凭空做到”的，而是应用提前暴露了可调用工具，例如查询 Prometheus、读取日志平台、调用发布系统 API、访问工单系统等。

## Agent 的核心组成

一个基础 Agent 通常包含五部分。

### 1. 大模型（Large Language Model，LLM）

大模型是 Agent 的“大脑”，负责理解用户输入、分析上下文、生成计划、选择工具、整理结果。

常见的大模型能力包括：

- 意图识别；
- 信息抽取；
- 任务拆解；
- 文本总结；
- 代码生成；
- 多轮推理；
- 工具参数生成。

但大模型本身不等于 Agent。没有工具、上下文和执行控制，大模型只能停留在问答层面。

### 2. 工具（Tools）

工具是 Agent 和外部世界交互的方式。

对后端开发来说，工具可以理解为一组被大模型调用的接口：

```text
queryOrder(orderId)
queryUser(userId)
searchDocs(keyword)
runSql(sql)
createTicket(title, content)
sendMessage(channel, content)
```

工具需要有清晰的名称、描述、入参、出参。大模型根据这些描述判断什么时候调用，以及应该传什么参数。

例如一个查询订单工具可以这样描述：

```json
{
  "name": "query_order",
  "description": "根据订单号查询订单基础信息、支付状态和物流状态",
  "parameters": {
    "type": "object",
    "properties": {
      "orderId": {
        "type": "string",
        "description": "订单号，例如 ORD202607050001"
      }
    },
    "required": ["orderId"]
  }
}
```

这段描述的重点不是 JSON 格式本身，而是让大模型知道：这个工具能解决什么问题、需要什么参数、返回结果代表什么含义。

### 3. 上下文（Context）

上下文决定 Agent 当前知道什么。

上下文可能来自：

- 用户当前输入；
- 历史对话；
- 系统提示词；
- 知识库检索结果；
- 数据库查询结果；
- 文件内容；
- 工具返回值；
- 当前环境信息。

上下文越准确，Agent 越容易做出合理判断。上下文混乱时，Agent 就可能误判任务、误用工具，甚至生成看似合理但实际错误的结论。

做业务系统时，不应该把所有数据都塞给模型，而是要根据任务动态补充必要上下文。

### 4. 记忆（Memory）

记忆负责保存任务执行过程中的关键信息。

短期记忆通常保存当前会话里的用户目标、工具调用结果、已经完成的步骤；长期记忆可以保存用户偏好、项目背景、常用配置等更稳定的信息。

记忆不是简单把所有聊天记录都塞回模型，而是要筛选出对当前任务有帮助的信息。记忆过多会增加成本，也可能引入无关噪声。

### 5. 执行器（Executor / Orchestrator）

执行器负责控制 Agent 的运行节奏。

它会处理这些问题：

- 一次任务最多执行多少步；
- 什么时候调用工具；
- 工具调用失败后是否重试；
- 哪些动作需要人工确认；
- 什么时候结束任务并返回结果。

如果说 LLM 负责“想下一步”，Tools 负责“做具体事”，Executor 就负责保证整个流程按规则运行，不让 Agent 无限循环或执行危险操作。

## Agent 的任务执行循环

很多 Agent 框架都会把执行过程设计成一个循环：观察、思考、行动、再观察。

图：AI Agent 任务执行循环示意图

![](images/2026/07/05/ai-agent-task-loop-placeholder.png)

可以用伪代码理解：

```python
def run_agent(user_goal):
    context = build_initial_context(user_goal)

    for step in range(MAX_STEPS):
        # 让大模型根据当前上下文判断下一步动作
        decision = llm.plan(context)

        if decision.type == "final_answer":
            return decision.content

        if decision.type == "tool_call":
            # 工具调用前要做参数校验和权限判断，不能完全相信模型输出
            checked_args = validate_args(decision.tool_name, decision.args)
            tool_result = call_tool(decision.tool_name, checked_args)

            # 工具结果追加进上下文，下一轮继续推理
            context.append({
                "tool": decision.tool_name,
                "args": checked_args,
                "result": tool_result
            })

    return "任务执行次数超过限制，请人工确认。"
```

这里有几个关键点：

- `MAX_STEPS` 必须存在，避免 Agent 陷入无限循环；
- 工具调用前必须校验参数，不能直接执行模型生成的内容；
- 工具返回值要进入上下文，下一轮才能继续判断；
- 最终回答应该基于真实工具结果，而不是只靠模型猜测。

## Agent 和 RAG 的区别

RAG 和 Agent 经常一起出现，但它们不是一回事。

RAG 主要解决“查资料再回答”的问题。

```text
用户提问
-> 检索知识库
-> 把相关资料交给大模型
-> 大模型生成回答
```

Agent 解决的是“围绕目标持续行动”的问题。

```text
用户提出目标
-> 规划步骤
-> 调用工具
-> 观察结果
-> 调整动作
-> 完成任务
```

RAG 可以成为 Agent 的一个工具。例如 Agent 需要查询内部文档时，可以调用知识库检索工具；拿到检索结果后，再决定是否继续调用其他工具。

所以更合理的关系是：

```text
Agent 负责调度任务
RAG 负责补充知识
工具负责执行动作
业务系统负责提供真实数据
```

## Agent 和工作流的区别

很多业务场景里，传统工作流已经能完成任务，比如审批流、定时任务、数据同步任务。

工作流的特点是路径固定：

```text
A 步骤 -> B 步骤 -> C 步骤 -> 结束
```

Agent 的特点是路径动态：

```text
根据当前结果判断下一步
可能调用工具 A
可能调用工具 B
也可能直接结束
```

如果业务流程高度确定，优先使用工作流，不必强行套 Agent。

适合 Agent 的场景通常有几个特点：

- 用户输入不固定；
- 任务步骤不完全固定；
- 需要阅读和理解大量文本；
- 需要根据中间结果调整动作；
- 需要跨多个系统收集信息；
- 最终输出需要自然语言总结。

不适合 Agent 的场景也很明确：

- 强事务一致性场景；
- 金额、库存、权限等高风险写操作；
- 固定规则就能完成的简单流程；
- 对结果可解释性和稳定性要求极高的核心链路。

## 一个最小 Agent 架构

后端项目里接入 Agent，不一定一开始就上复杂框架。可以先按下面的结构拆分：

```text
agent-service
├── controller        接收用户请求
├── planner           调用大模型，生成下一步动作
├── tool-registry     管理可用工具列表
├── tools             具体工具实现
├── memory            会话历史和任务上下文
├── guardrail         权限、参数、安全规则
└── executor          执行循环控制
```

核心链路可以这样设计：

```text
用户请求
-> Controller 创建任务
-> Planner 生成工具调用计划
-> Guardrail 做权限和参数检查
-> Executor 调用 Tool
-> Tool 返回业务结果
-> Memory 记录上下文
-> Planner 判断是否继续
-> 返回最终结果
```

如果工具需要标准化接入，可以了解 MCP。它的定位是让大模型应用以统一方式连接外部工具、资源和提示词。MCP 官方网站：[Model Context Protocol](https://modelcontextprotocol.io/)。之前整理过一篇基础说明：[[posts/MCP 入门理解：大模型如何通过工具调用连接外部系统]]。

## 工具设计比模型选择更容易被忽视

Agent 能不能稳定工作，很多时候不只看模型能力，还要看工具设计。

一个糟糕的工具可能是这样的：

```text
execute(command)
```

它看起来很强，什么都能执行，但风险也很高。模型一旦生成危险命令，就可能造成严重后果。

更稳妥的设计是把工具拆细：

```text
query_service_metrics(serviceName, startTime, endTime)
query_error_logs(serviceName, keyword, startTime, endTime)
query_deploy_records(serviceName, startTime, endTime)
create_incident_ticket(title, content, level)
```

工具越清晰，Agent 越容易正确选择；权限越明确，系统越安全。

设计工具时可以注意几点：

- 名称表达具体能力，不要过度抽象；
- 描述写清楚适用场景；
- 参数类型尽量简单明确；
- 返回结果保持结构化；
- 写操作必须有权限控制；
- 高风险动作增加人工确认；
- 工具执行失败时返回明确错误原因。

## 记忆不等于把所有历史都保存下来

Agent 经常会提到 memory，也就是记忆。

记忆大致可以分为两类：

```text
短期记忆：当前任务中的上下文、工具结果、对话历史
长期记忆：用户偏好、项目背景、业务知识、历史处理记录
```

短期记忆适合放在一次任务上下文里，任务结束后可以归档。

长期记忆需要更谨慎，尤其是涉及隐私、权限和业务数据时。不是所有历史对话都适合长期保存，也不是所有保存下来的内容都应该在后续任务里自动注入。

比较稳妥的做法是：

- 明确哪些信息允许记忆；
- 敏感信息不进入长期记忆；
- 记忆内容可查看、可删除；
- 注入上下文前做权限判断；
- 长期记忆尽量结构化。

## 落地时最常见的几个坑

### 工具权限过大

让 Agent 直接执行 SQL、Shell 或生产操作非常危险。即使模型表现再好，也不能绕过权限系统。

高风险工具至少需要做到：

- 只读和写操作分离；
- 生产环境写操作二次确认；
- 操作日志可追踪；
- 参数白名单校验；
- 用户权限和工具权限绑定。

### 任务边界不清楚

用户说“帮我优化一下系统”，这个目标太大，Agent 很难判断该查哪里、查到什么程度、什么时候结束。

更好的方式是引导用户把目标收敛：

```text
帮我分析订单服务最近 1 小时 P95 响应时间升高的原因，只需要只读排查，不执行变更操作。
```

目标越具体，执行路径越稳定。

### 过度依赖模型推理

模型不是规则引擎。关键业务规则应该写在代码里，而不是期待模型每次都“想明白”。

比如：

```text
退款金额不能大于支付金额
普通客服不能查看完整身份证号
生产环境不能自动执行删除命令
```

这些规则必须由系统强制校验。

### 缺少执行过程记录

Agent 做了哪些判断、调用了哪些工具、工具返回了什么、为什么给出这个结论，都应该有记录。

没有执行轨迹时，一旦结果出错，很难排查问题。

## 后端开发怎么开始学习 Agent

可以按这个顺序入门：

1. 先理解大模型 API 的基本调用方式；
2. 学会 Function Calling / Tool Calling 的参数描述方式；
3. 写一个只读工具，例如查询订单、查询天气、检索文档；
4. 给工具加参数校验和权限判断；
5. 把工具返回值加入上下文，让模型继续总结；
6. 增加执行步数限制和错误处理；
7. 再考虑记忆、RAG、MCP、任务编排等能力。

可以先做一个很小的练习：

```text
输入：查询某个订单号的状态
工具：query_order(orderId)
输出：订单状态、支付状态、物流状态的自然语言总结
```

这个练习覆盖了 Agent 最基础的几个环节：理解用户意图、抽取参数、调用工具、读取结果、生成回答。

## 最后做个简单总结

AI Agent 可以理解为“大模型 + 工具 + 上下文 + 执行控制”的组合。

大模型负责理解和规划，工具负责执行真实动作，上下文负责提供任务信息，执行控制负责保证流程不会失控。

真正落地时，重点把工具边界、权限校验、上下文管理、执行日志这些工程细节做好。对后端开发来说，Agent 并不神秘，它更像是一个会调用接口、会读取结果、会继续判断下一步的智能调度层。
