## 前言

随着大语言模型逐渐进入搜索、编程、办公和企业业务系统，Agent 成了 AI 应用开发中经常出现的概念。

很多人第一次接触 Agent 时，会产生一些疑问：Agent 是不是一个更强的大模型？是不是只要调用了工具就算 Agent？它和传统工作流、后端服务又有什么区别？

这一篇先不急着使用 LangChain 等框架，而是从 Agent 的定义、能力和运行机制入手，理解一个 Agent 到底是怎样围绕目标持续执行任务的。

## 什么是 Agent

AI Agent 通常被翻译为“智能体”。可以把它定义为：

> AI Agent 是一个以大语言模型为核心决策模块，在给定目标、工具和约束条件下，能够读取上下文、规划下一步动作、调用外部工具，并根据执行结果持续调整的任务执行系统。

这一定义中有几个关键词：

- **目标**：Agent 需要知道最终要解决什么问题；
- **上下文**：包括用户输入、历史对话、工具结果和外部环境状态；
- **规划**：模型根据当前状态判断下一步应该做什么；
- **工具**：Agent 通过搜索、数据库、API 或代码执行连接外部世界；
- **反馈**：工具执行结果会重新进入上下文；
- **迭代**：如果任务还没有完成，Agent 会继续思考和行动。

因此，Agent 不是一个全新的模型，而是一种围绕大语言模型构建的任务执行系统。模型负责理解和决策，工具负责执行真实操作，程序则负责管理上下文、权限、状态和执行循环。

## Agent 具备哪些能力

普通 LLM 调用通常是“一次输入、一次输出”，而 Agent 更强调多轮自主执行。

| 维度 | 普通 LLM 调用 | AI Agent |
| --- | --- | --- |
| 交互模式 | 根据输入直接生成一次结果 | 接收目标后自主规划、执行和迭代 |
| 工具使用 | 通常不使用工具，或由程序固定调用 | 根据任务动态选择搜索、数据库、API、代码执行等工具 |
| 环境感知 | 主要依赖当前提示词 | 感知对话、工具结果和外部环境状态 |
| 任务复杂度 | 适合翻译、摘要、改写等单步任务 | 适合调研、分析、排查、报告生成等多步任务 |
| 状态管理 | 通常不维护任务状态 | 维护工作记忆、长期记忆和任务进度 |
| 错误处理 | 一次调用失败后通常由外部程序处理 | 可以重试、切换工具或调整原有计划 |

Agent 的核心价值，是处理那些需要**多步推理、工具协作和动态决策**的复杂任务。

例如，用户提出：

```text
帮我分析订单服务最近半小时大量超时的原因，并整理一份排查结论。
```

Agent 可能会依次执行：

```text
1. 查询接口耗时指标
2. 查询服务器 CPU、内存和 JVM GC 情况
3. 检索错误日志与慢 SQL
4. 查询最近的服务发布记录
5. 对比不同数据的时间点
6. 汇总可能原因并生成报告
```

执行过程中，如果监控数据正常但数据库出现大量慢查询，Agent 就可以调整原计划，把后续排查重点放到数据库和 SQL 上。

但这不意味着所有场景都应该使用 Agent。如果只是翻译一段文字、生成一个标题或总结一小段内容，一次普通 LLM 调用通常更快、更便宜，也更加稳定。

## Agent 的核心架构：TAO 循环

为了方便理解，可以把 Agent 的核心运行过程概括为 TAO 循环：

```text
Think（思考） -> Act（行动） -> Observe（观察）
       ^                              |
       |______________________________|
```

无论 Agent 外部接入了多少工具，内部增加了多少记忆或工作流，它的核心都可以抽象为：分析当前状态、执行一个动作、观察执行结果，然后决定下一步。

### Think：思考

LLM 在这一阶段充当 Agent 的“大脑”，根据用户目标和当前上下文判断下一步行动。

它可能需要完成这些判断：

- 当前任务是否已经完成；
- 是否缺少完成任务所需的信息；
- 下一步需要调用哪个工具；
- 工具调用参数应该是什么；
- 是否需要先执行其他步骤；
- 当前操作是否存在风险。

例如，用户询问“LangChain 最新版本的 Agent 怎么使用”，模型应该意识到“最新版本”属于动态信息，仅依靠模型已有知识可能不够可靠，需要先查询官方文档。

### Act：行动

Agent 根据 Think 阶段的决策执行动作。

动作通常有两种：

1. 调用工具，例如搜索文档、读取文件、查询数据库或执行代码；
2. 任务已经完成，直接生成最终回答。

如果 Agent 决定查询 LangChain 官方文档，Act 阶段就会向搜索工具或文档工具传入关键词并执行查询。

需要注意的是，模型只负责生成“调用什么工具、传入什么参数”的决策。真正的 API 请求、数据库查询或代码执行，仍然由外部程序完成。

### Observe：观察

Observe 阶段负责收集行动结果。

如果上一阶段调用了工具，就把工具返回的数据加入任务上下文；如果已经向用户输出了结果，也可以根据用户反馈判断任务是否需要继续。

可以把新的上下文简化理解为：

```text
新上下文 = 用户原始问题 + 历史执行过程 + 最新工具结果
```

模型在下一轮 Think 阶段读取这些信息。如果现有资料已经足够，就生成最终回答；如果信息仍然不足，就继续调用其他工具。

## 用一个完整过程理解 TAO 循环

仍然以查询 LangChain 最新用法为例：

```text
用户：LangChain 最新版本的 Agent 怎么使用？
```

Agent 的执行过程可能是：

```text
第 1 轮 Think
问题涉及“最新版本”，模型已有知识可能过期，需要查询官方文档。

第 1 轮 Act
调用文档搜索工具，检索 LangChain 官方 Agent 文档。

第 1 轮 Observe
获得当前版本的安装方式、核心 API 和代码示例，并把结果加入上下文。

第 2 轮 Think
资料已经足够，不需要继续调用工具，可以整理答案。

第 2 轮 Act
基于官方文档生成最终回答，并附上来源。
```

用伪代码表示，大致如下：

```python
def run_agent(user_goal):
    context = [user_goal]

    for step in range(MAX_STEPS):
        decision = llm.think(context)

        if decision.type == "final_answer":
            return decision.content

        if decision.type == "tool_call":
            result = call_tool(
                decision.tool_name,
                decision.arguments
            )
            context.append(result)

    return "任务执行步数超过限制，请人工确认。"
```

TAO 循环的重要特点是**能够自行判断是否终止**。Agent 在每一轮 Think 阶段都会判断任务是否完成：完成就输出最终结果，未完成就进入下一轮循环。

因此，Agent 可以根据任务复杂度动态调整执行步数。简单任务可能一轮结束，复杂任务则可能执行五轮、十轮甚至更多。工程实现中仍然需要设置最大执行步数、超时时间和调用费用限制，避免 Agent 陷入无效循环。

## Agent、普通 LLM 与 Workflow 的区别

### Agent 是不是 ChatGPT

不是同一个概念。

ChatGPT 是一个面向用户的大语言模型应用，而 Agent 是一种围绕目标执行任务的系统设计方式。一个聊天应用可以加入 Agent 能力，一个 Agent 也可以使用不同厂商的大语言模型作为决策模块。

可以简单理解为：

```text
普通聊天：你问一句，模型答一句
Agent：你给出目标，它拆分步骤、调用工具、观察结果，然后继续执行
```

### Agent 和 Workflow 有什么区别

Workflow 的执行步骤通常由开发者提前规定：

```text
步骤 A -> 步骤 B -> 步骤 C -> 结束
```

Agent 则由开发者提供目标、工具和约束，再由模型根据当前状态动态决定下一步：

```text
当前结果 -> 模型判断 -> 工具 A / 工具 B / 直接结束
```

如果业务路径固定、规则明确，Workflow 通常更合适；如果输入不固定，而且需要根据中间结果不断调整执行路径，Agent 更有优势。

在真实项目中，两者也可以结合：整体流程由 Workflow 保证稳定性，某个需要自然语言理解和动态决策的节点交给 Agent 处理。

## 新手常见问题

### Q1：Agent 一定要调用工具吗

不一定。

如果任务很简单，Agent 可以判断不需要工具并直接回答。但工具是 Agent 扩展能力的关键。没有工具时，它的能力主要来自模型和已有上下文；接入工具后，它才能搜索实时资料、查询数据库、读取文件、执行代码或调用业务 API。

### Q2：只要调用工具就是 Agent 吗

不一定。

如果程序始终按照固定顺序调用工具，它更接近一个带 LLM 节点的工作流。Agent 更强调模型能根据目标和中间结果动态决定下一步，并形成“思考—行动—观察”的执行闭环。

### Q3：Agent 一定比普通 LLM 更强吗

不是。

Agent 适合复杂任务，但也会带来额外成本：

- 执行时间更长；
- 模型和工具调用费用更高；
- 执行路径存在不确定性；
- 错误可能在多轮执行中累积；
- 调试和效果评估更加困难。

如果一次 LLM 调用就能解决问题，就没有必要强行设计成 Agent。

### Q4：学习 Agent 需要先掌握很多框架吗

不需要。

LangChain、LangGraph 等框架可以提高开发效率，但框架不是 Agent 的本质。初学时更重要的是先理解这条主线：

```text
目标 -> 思考 -> 行动 -> 观察 -> 再思考 -> 最终回答
```

理解这个循环后，再去学习不同框架里的 Agent、Tool、Message、Memory 和 Executor，就会容易很多。

### Q5：Agent 出错后为什么能重试

因为错误信息也可以作为 Observe 阶段的一部分加入上下文。

例如数据库查询工具返回“字段不存在”，模型可以阅读错误信息，修改查询参数后再次调用；如果某个搜索工具不可用，也可以选择其他工具。

但重试能力必须受到程序约束，例如限制重试次数、区分可重试错误和不可重试错误，不能完全交给模型自由决定。

## Agent 和企业后端服务是什么关系

可以先做一个简单区分：

```text
后端服务：负责确定性的业务逻辑
Agent：负责不确定性较高的理解、决策和任务执行
```

Agent 通常是企业系统的一部分，而不是用来替代整个后端系统。后端仍然需要负责接口、权限、数据、事务和业务规则，Agent 则在这些约束下调用工具完成任务。

一种常见架构是：

```text
前端 -> 后端服务 -> Agent -> 工具 / API / 数据库
```

例如用户提出：

```text
帮我查一下这个客户最近三个月的订单异常情况。
```

后端服务负责：

1. 校验当前用户是否有权查看该客户的数据；
2. 提供受控的订单查询接口；
3. 提供明确的异常订单判定规则；
4. 对敏感字段进行脱敏；
5. 记录工具调用和用户操作日志。

Agent 负责：

1. 理解用户具体想查询什么；
2. 判断需要调用哪些订单接口；
3. 根据接口结果决定是否补充查询；
4. 分析异常订单的共同特征；
5. 总结异常原因并生成自然语言结果。

权限校验、金额计算、事务处理等确定性逻辑不能依赖模型临时判断，而应该由后端代码强制执行。Agent 可以决定“要不要调用某个工具”，但不能绕过工具本身的权限和业务规则。

## 什么场景适合使用 Agent

比较适合 Agent 的任务通常具有以下特点：

- 用户目标明确，但具体执行步骤不固定；
- 需要跨多个系统查询和整理信息；
- 需要根据中间结果动态调整计划；
- 输入和输出以自然语言或非结构化数据为主；
- 允许通过校验、重试和人工确认处理一定的不确定性。

不适合直接交给 Agent 的任务包括：

- 固定规则就能完成的简单流程；
- 强事务一致性的核心交易链路；
- 未经确认的高风险生产操作；
- 对结果稳定性和可重复性要求极高的计算；
- 权限、金额、库存等必须严格校验的业务逻辑。

判断是否需要 Agent 时，可以先问自己一个问题：

> 这个任务是否真的需要模型根据中间结果动态决定下一步？

如果答案是否定的，普通 LLM 调用或传统 Workflow 往往是更简单、更可靠的方案。

## 总结

Agent 不是一个新的大模型，而是一个以大语言模型为决策核心，结合上下文、工具、状态和执行控制构建的任务执行系统。

它的核心运行机制可以概括为 TAO 循环：

```text
Think：分析目标和当前状态，决定下一步
Act：调用工具或生成最终回答
Observe：收集执行结果并更新上下文
```

这个循环会持续运行，直到 Agent 判断任务已经完成。Agent 的优势是处理多步、动态、需要工具协作的复杂任务，但它并不天然优于普通 LLM 或 Workflow。

学习 Agent 时，不必一开始就陷入复杂框架。先理解“目标—思考—行动—观察—再思考”这条主线，再通过代码实现一个最小执行循环，才能真正理解 Agent 是如何工作的。
