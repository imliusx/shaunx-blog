## 背景

订单创建后，用户通常需要在固定时间内完成支付。例如：

```text
用户提交订单
-> 系统锁定库存
-> 用户 30 分钟内支付
-> 支付成功后订单进入已支付状态
-> 超过 30 分钟未支付，系统自动取消订单并释放库存
```

这个功能看起来只是“30 分钟后取消订单”，实际会牵涉到很多细节：

- 延迟任务如何触发；
- 订单状态如何防止并发错乱；
- 用户支付和系统取消同时发生怎么办；
- 消息丢失、服务重启、任务失败如何补偿；
- 库存、优惠券、积分等资源如何回滚；
- 高并发订单场景下如何减少数据库压力。

订单超时取消不能只靠一段延迟代码解决，更需要完整的状态设计、幂等设计和补偿机制。

## 业务目标

一个稳定的超时取消方案，至少要满足这些目标：

| 目标 | 说明 |
| --- | --- |
| 准确性 | 只有未支付且已超时的订单才能被取消 |
| 幂等性 | 延迟消息重复消费、定时任务重复扫描都不能导致重复释放库存 |
| 可恢复 | MQ 消息丢失、消费者异常、服务重启后仍能补偿处理 |
| 可观测 | 能看到待取消数量、取消成功数、失败数、积压量 |
| 可扩展 | 支持未来接入优惠券、积分、库存、风控、通知等业务 |
| 性能可控 | 避免全表扫描和大量无意义查询 |

订单超时取消的核心不是“定时触发”，而是“最终一定能把该取消的订单安全地取消掉”。

## 状态模型

订单状态先要定义清楚，否则后续逻辑很容易混乱。

常见状态可以这样设计：

| 状态 | 含义 | 是否可取消 |
| --- | --- | --- |
| `UNPAID` | 待支付 | 是 |
| `PAID` | 已支付 | 否 |
| `CANCELED_TIMEOUT` | 超时取消 | 否 |
| `CANCELED_USER` | 用户主动取消 | 否 |
| `CLOSED` | 关闭 | 否 |
| `REFUNDED` | 已退款 | 否 |

超时取消只允许处理 `UNPAID` 状态的订单。

状态流转可以简化为：

```text
UNPAID -> PAID
UNPAID -> CANCELED_TIMEOUT
UNPAID -> CANCELED_USER
PAID -> REFUNDED
```

不建议让订单状态随意跳转。每一次状态变更都应该有明确的来源和条件。

## 表结构

订单表里建议保存过期时间，而不是每次用创建时间动态计算。

```sql
CREATE TABLE `orders` (
  `id` BIGINT NOT NULL PRIMARY KEY,
  `order_no` VARCHAR(64) NOT NULL,
  `user_id` BIGINT NOT NULL,
  `status` VARCHAR(32) NOT NULL,
  `expire_time` DATETIME NOT NULL,
  `cancel_type` VARCHAR(32) DEFAULT NULL,
  `cancel_time` DATETIME DEFAULT NULL,
  `version` INT NOT NULL DEFAULT 0,
  `created_at` DATETIME NOT NULL,
  `updated_at` DATETIME NOT NULL,
  UNIQUE KEY `uk_order_no` (`order_no`),
  KEY `idx_timeout_order` (`status`, `expire_time`, `id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

`idx_timeout_order` 很关键，定时补偿任务会用到：

```sql
SELECT id
FROM orders
WHERE status = 'UNPAID'
  AND expire_time <= NOW()
ORDER BY id
LIMIT 200;
```

没有这个索引，高频补偿任务很容易变成全表扫描。

如果系统需要记录取消过程，也可以增加一张任务表：

```sql
CREATE TABLE `order_timeout_task` (
  `id` BIGINT NOT NULL PRIMARY KEY,
  `order_id` BIGINT NOT NULL,
  `execute_time` DATETIME NOT NULL,
  `status` VARCHAR(32) NOT NULL,
  `retry_count` INT NOT NULL DEFAULT 0,
  `last_error` VARCHAR(512) DEFAULT NULL,
  `created_at` DATETIME NOT NULL,
  `updated_at` DATETIME NOT NULL,
  UNIQUE KEY `uk_order_id` (`order_id`),
  KEY `idx_task_execute` (`status`, `execute_time`, `id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

任务表不是必须的。订单量不大时，直接扫描订单表即可；订单量大、需要审计和重试记录时，任务表更清晰。

## 方案对比

常见实现方式有四种。

| 方案 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| MQ 延迟消息 | 实时性好，削峰能力强，支持重试 | 依赖消息中间件，需要处理重复消费 | 主流推荐方案 |
| Redis 过期监听 | 实现简单，触发较快 | 事件可能丢失，不适合作为强可靠核心链路 | 低风险提醒、辅助通知 |
| 定时任务轮询 | 简单可靠，容易补偿 | 实时性较弱，需要控制扫描压力 | 兜底补偿、低订单量系统 |
| 时间轮 | 性能高，触发精度高 | 内存任务重启丢失，集群一致性复杂 | 单机高性能延迟任务、框架内部调度 |

更稳妥的选择是：

```text
MQ 延迟消息负责主要触发
定时任务负责兜底补偿
数据库条件更新保证幂等
监控告警保证问题可发现
```

## MQ 延迟消息

订单创建成功后，发送一条延迟消息。消息到期后，消费者检查订单状态，如果仍然是未支付且已经超时，就执行取消逻辑。

RabbitMQ 可以通过 TTL + 死信交换机实现延迟队列，相关文档可以查看 RabbitMQ 的 [TTL](https://www.rabbitmq.com/docs/ttl) 和 [Dead Letter Exchanges](https://www.rabbitmq.com/docs/dlx)。如果使用 RocketMQ，可以直接使用延迟消息，RocketMQ 延迟消息说明：[Delay Message](https://rocketmq.apache.org/docs/featureBehavior/02delaymessage/)。

### RabbitMQ 配置

下面是一个固定 30 分钟超时的 RabbitMQ 配置示例。

```java
@Configuration
public class OrderTimeoutRabbitConfig {

    public static final String DELAY_EXCHANGE = "order.timeout.delay.exchange";
    public static final String DELAY_QUEUE = "order.timeout.delay.queue";
    public static final String DELAY_ROUTING_KEY = "order.timeout.delay";

    public static final String TIMEOUT_EXCHANGE = "order.timeout.exchange";
    public static final String TIMEOUT_QUEUE = "order.timeout.queue";
    public static final String TIMEOUT_ROUTING_KEY = "order.timeout";

    @Bean
    public DirectExchange orderTimeoutDelayExchange() {
        return new DirectExchange(DELAY_EXCHANGE, true, false);
    }

    @Bean
    public DirectExchange orderTimeoutExchange() {
        return new DirectExchange(TIMEOUT_EXCHANGE, true, false);
    }

    @Bean
    public Queue orderTimeoutDelayQueue() {
        return QueueBuilder.durable(DELAY_QUEUE)
                // 队列里的消息存活 30 分钟，到期后进入死信交换机
                .ttl(30 * 60 * 1000)
                .deadLetterExchange(TIMEOUT_EXCHANGE)
                .deadLetterRoutingKey(TIMEOUT_ROUTING_KEY)
                .build();
    }

    @Bean
    public Queue orderTimeoutQueue() {
        // 真正被消费者监听的队列
        return QueueBuilder.durable(TIMEOUT_QUEUE).build();
    }

    @Bean
    public Binding delayBinding() {
        return BindingBuilder.bind(orderTimeoutDelayQueue())
                .to(orderTimeoutDelayExchange())
                .with(DELAY_ROUTING_KEY);
    }

    @Bean
    public Binding timeoutBinding() {
        return BindingBuilder.bind(orderTimeoutQueue())
                .to(orderTimeoutExchange())
                .with(TIMEOUT_ROUTING_KEY);
    }
}
```

如果每个订单的过期时间不一样，不建议在同一个普通队列里混用大量不同的消息 TTL，可能遇到队头阻塞问题。可以考虑 RabbitMQ 延迟消息插件、RocketMQ 延迟消息，或者把不同延迟时间拆成不同队列。

### 发送消息

订单创建成功后再发送延迟消息，避免出现“消息发送成功，但订单事务回滚”的问题。

```java
@Service
public class OrderCreateService {

    private final OrderRepository orderRepository;
    private final OrderTimeoutProducer orderTimeoutProducer;

    public OrderCreateService(OrderRepository orderRepository,
                              OrderTimeoutProducer orderTimeoutProducer) {
        this.orderRepository = orderRepository;
        this.orderTimeoutProducer = orderTimeoutProducer;
    }

    @Transactional(rollbackFor = Exception.class)
    public Long createOrder(CreateOrderCommand command) {
        LocalDateTime now = LocalDateTime.now();
        LocalDateTime expireTime = now.plusMinutes(30);

        Order order = new Order();
        order.setId(IdGenerator.nextId());
        order.setOrderNo(OrderNoGenerator.next());
        order.setUserId(command.getUserId());
        order.setStatus(OrderStatus.UNPAID.name());
        order.setExpireTime(expireTime);
        order.setCreatedAt(now);
        order.setUpdatedAt(now);

        // 订单入库和库存锁定应该在同一个本地事务内完成
        orderRepository.insert(order);
        orderRepository.lockStock(order.getId(), command.getItems());

        TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() {
            @Override
            public void afterCommit() {
                // 只有订单事务提交成功后，才发送超时取消消息
                orderTimeoutProducer.send(order.getId(), order.getExpireTime());
            }
        });

        return order.getId();
    }
}
```

如果对消息可靠性要求更高，可以使用本地消息表或 Outbox 模式：订单事务里先写入一条待发送消息，再由后台任务投递到 MQ。

### 消费消息

消费者只负责触发取消，真正的幂等判断放到领域服务里。

```java
@Component
public class OrderTimeoutConsumer {

    private final OrderTimeoutService orderTimeoutService;

    public OrderTimeoutConsumer(OrderTimeoutService orderTimeoutService) {
        this.orderTimeoutService = orderTimeoutService;
    }

    @RabbitListener(queues = OrderTimeoutRabbitConfig.TIMEOUT_QUEUE)
    public void onMessage(OrderTimeoutMessage message) {
        // MQ 可能重复投递，取消逻辑必须支持幂等
        orderTimeoutService.cancelIfTimeout(message.getOrderId(), "MQ_DELAY_MESSAGE");
    }
}
```

核心取消逻辑：

```java
@Service
public class OrderTimeoutService {

    private final OrderRepository orderRepository;
    private final StockRepository stockRepository;
    private final ApplicationEventPublisher eventPublisher;

    public OrderTimeoutService(OrderRepository orderRepository,
                               StockRepository stockRepository,
                               ApplicationEventPublisher eventPublisher) {
        this.orderRepository = orderRepository;
        this.stockRepository = stockRepository;
        this.eventPublisher = eventPublisher;
    }

    @Transactional(rollbackFor = Exception.class)
    public void cancelIfTimeout(Long orderId, String source) {
        LocalDateTime now = LocalDateTime.now();

        int updated = orderRepository.cancelTimeoutOrder(
                orderId,
                OrderStatus.UNPAID.name(),
                OrderStatus.CANCELED_TIMEOUT.name(),
                now
        );

        if (updated == 0) {
            // 可能已经支付、已经取消、还没到过期时间，直接返回即可
            return;
        }

        // 只有订单状态成功变更后，才释放库存，避免重复释放
        stockRepository.releaseLockedStock(orderId);

        // 发布订单取消事件，优惠券、积分、通知等逻辑可以异步订阅
        eventPublisher.publishEvent(new OrderCanceledEvent(orderId, source));
    }
}
```

对应 SQL 要写成条件更新：

```sql
UPDATE orders
SET status = 'CANCELED_TIMEOUT',
    cancel_type = 'TIMEOUT',
    cancel_time = NOW(),
    updated_at = NOW(),
    version = version + 1
WHERE id = #{orderId}
  AND status = 'UNPAID'
  AND expire_time <= NOW();
```

这条 SQL 是幂等控制的关键。重复执行时，只有第一次能更新成功，后续都会返回 `0`。

## 定时补偿

延迟消息不是百分百兜底。可能发生这些情况：

- 消息发送失败；
- MQ 短时间不可用；
- 消费者异常；
- 消息进入死信队列后无人处理；
- 人工修复数据导致状态遗漏。

所以必须有定时补偿任务，定期扫描已经超时但仍然未支付的订单。

```java
@Component
public class OrderTimeoutCompensateJob {

    private static final int BATCH_SIZE = 200;

    private final OrderRepository orderRepository;
    private final OrderTimeoutService orderTimeoutService;

    public OrderTimeoutCompensateJob(OrderRepository orderRepository,
                                     OrderTimeoutService orderTimeoutService) {
        this.orderRepository = orderRepository;
        this.orderTimeoutService = orderTimeoutService;
    }

    @Scheduled(cron = "0 */5 * * * ?")
    public void compensate() {
        LocalDateTime now = LocalDateTime.now();
        Long lastId = 0L;

        while (true) {
            // 按 id 分页，避免 offset 深分页带来的性能问题
            List<Long> orderIds = orderRepository.findTimeoutUnpaidOrderIds(now, lastId, BATCH_SIZE);
            if (orderIds.isEmpty()) {
                break;
            }

            for (Long orderId : orderIds) {
                try {
                    orderTimeoutService.cancelIfTimeout(orderId, "SCHEDULE_COMPENSATE");
                } catch (Exception ex) {
                    // 单个订单失败不能影响整批任务，生产环境需要记录失败日志和告警
                    log.error("订单超时取消补偿失败，orderId={}", orderId, ex);
                }
            }

            lastId = orderIds.get(orderIds.size() - 1);
        }
    }
}
```

查询 SQL：

```sql
SELECT id
FROM orders
WHERE status = 'UNPAID'
  AND expire_time <= #{now}
  AND id > #{lastId}
ORDER BY id
LIMIT #{limit};
```

定时任务频率不需要太高。订单超时取消通常允许几十秒到几分钟误差，5 分钟扫一次已经能覆盖大部分兜底场景。

## 并发问题

### 支付与取消并发

最典型的问题是：用户刚好在第 30 分钟支付，同时系统触发超时取消。

不能只靠应用层判断：

```java
if (order.getStatus() == UNPAID) {
    cancel(order);
}
```

这种写法在并发下不安全。正确做法是让数据库条件更新决定谁成功。

支付成功回调可以这样更新：

```sql
UPDATE orders
SET status = 'PAID',
    updated_at = NOW(),
    version = version + 1
WHERE id = #{orderId}
  AND status = 'UNPAID';
```

超时取消也使用条件更新：

```sql
UPDATE orders
SET status = 'CANCELED_TIMEOUT',
    cancel_type = 'TIMEOUT',
    cancel_time = NOW(),
    updated_at = NOW(),
    version = version + 1
WHERE id = #{orderId}
  AND status = 'UNPAID'
  AND expire_time <= NOW();
```

谁先更新成功，谁获得订单状态流转权。

如果支付回调发现更新失败，需要重新查询订单状态：

- 已支付：重复回调，直接返回成功；
- 已超时取消：根据业务规则发起退款或关闭支付单；
- 用户取消：关闭支付单或退款；
- 其他状态：记录异常并进入人工排查。

### 库存重复释放

库存释放也要幂等。

如果库存和订单在同一个库，可以用订单状态更新成功作为释放前提。只有 `updated == 1` 时才释放库存。

如果库存是独立服务，建议通过订单取消事件通知库存服务，并在库存服务侧做幂等：

```text
订单服务取消成功
-> 发送 OrderCanceledEvent
-> 库存服务消费事件
-> 根据 orderId 判断是否已经释放
-> 未释放则执行释放并记录释放流水
```

库存释放流水可以用 `order_id` 做唯一键，防止重复消费导致库存加多。

## Redis 过期监听

Redis 过期监听实现起来很简单：订单创建时写一个带过期时间的 key，key 到期后监听过期事件。

Redis Keyspace Notifications 文档：[Redis Keyspace notifications](https://redis.io/docs/latest/develop/pubsub/keyspace-notifications/)。

```java
redisTemplate.opsForValue().set(
        "order:timeout:" + orderId,
        "1",
        30,
        TimeUnit.MINUTES
);
```

监听过期事件：

```java
@Component
public class OrderRedisExpiredListener {

    private final OrderTimeoutService orderTimeoutService;

    public OrderRedisExpiredListener(OrderTimeoutService orderTimeoutService) {
        this.orderTimeoutService = orderTimeoutService;
    }

    @EventListener
    public void onExpiredKey(RedisKeyExpiredEvent<String> event) {
        String expiredKey = event.getValue();
        if (!expiredKey.startsWith("order:timeout:")) {
            return;
        }

        // key 格式：order:timeout:{orderId}
        Long orderId = Long.valueOf(expiredKey.substring("order:timeout:".length()));
        orderTimeoutService.cancelIfTimeout(orderId, "REDIS_EXPIRED_EVENT");
    }
}
```

这个方案不建议作为核心方案，原因是 Redis 过期事件更像通知，不是可靠消息：

- 服务宕机期间可能错过事件；
- Redis 配置不正确时不会发送事件；
- 主从切换时可能出现事件丢失；
- 高峰期过期触发时间不一定精确。

如果使用 Redis 过期监听，一定要配合定时任务补偿。

## 时间轮

时间轮适合管理大量延迟任务。Netty 提供了 `HashedWheelTimer`，API 文档：[HashedWheelTimer](https://netty.io/4.1/api/io/netty/util/HashedWheelTimer.html)。

```java
HashedWheelTimer timer = new HashedWheelTimer(
        1,
        TimeUnit.SECONDS,
        3600
);

public void submitTimeoutTask(Long orderId) {
    timer.newTimeout(timeout -> {
        // 时间轮只负责触发，最终仍然依赖数据库条件更新保证幂等
        orderTimeoutService.cancelIfTimeout(orderId, "HASHED_WHEEL_TIMER");
    }, 30, TimeUnit.MINUTES);
}
```

时间轮的性能很好，但要注意边界：

- 任务在内存中，服务重启后任务会丢失；
- 多实例部署时需要考虑任务分配；
- 不适合直接承担强可靠订单取消职责；
- 更适合网关超时、连接管理、短周期内部任务。

订单系统里可以借鉴时间轮思想，但最终仍建议把订单过期时间落库，并配合补偿扫描。

## 推荐架构

推荐使用“延迟消息 + 定时补偿 + 条件更新”的组合。

图：延迟消息与定时补偿架构图

![](images/2026/07/05/order-timeout-mq-compensation-placeholder.png)

核心链路：

```text
创建订单
-> 写入订单，状态为 UNPAID，保存 expire_time
-> 锁定库存
-> 事务提交后发送延迟消息
-> 延迟消息到期，消费者触发取消
-> 数据库条件更新订单状态
-> 更新成功后释放库存
-> 发布订单取消事件
-> 定时任务补偿漏处理订单
```

这个方案的好处是：

- MQ 负责大部分实时触发；
- 定时任务负责最终兜底；
- 数据库条件更新负责并发安全；
- 事件机制负责扩展库存、优惠券、通知等后续动作；
- 监控告警负责及时发现异常。

## 监控告警

超时取消功能上线后，要重点关注这些指标：

| 指标 | 说明 |
| --- | --- |
| `order_timeout_message_send_count` | 延迟消息发送数量 |
| `order_timeout_message_consume_count` | 延迟消息消费数量 |
| `order_timeout_cancel_success_count` | 超时取消成功数量 |
| `order_timeout_cancel_skip_count` | 已支付或已取消导致跳过数量 |
| `order_timeout_cancel_fail_count` | 取消失败数量 |
| `order_timeout_compensate_count` | 定时补偿处理数量 |
| `order_timeout_pending_count` | 已超时但仍未支付的订单数量 |
| `order_timeout_message_lag` | 延迟队列积压或消费延迟 |

告警规则可以先设置得简单一些：

- 已超时未支付订单数量持续增加；
- 取消失败数超过阈值；
- 延迟队列积压超过阈值；
- 补偿任务连续失败；
- 死信队列出现大量消息。

只有业务逻辑没有监控，出问题时很难及时发现。

## 实施清单

落地时可以按这个顺序推进：

1. 订单表增加 `expire_time`、`cancel_type`、`cancel_time` 字段；
2. 增加 `status + expire_time + id` 组合索引；
3. 创建订单时写入过期时间；
4. 订单事务提交后发送延迟消息；
5. 消费者调用统一的 `cancelIfTimeout` 方法；
6. 取消 SQL 使用 `status = UNPAID AND expire_time <= NOW()` 条件；
7. 库存释放、优惠券回滚、积分回滚都要做幂等；
8. 增加定时补偿任务；
9. 增加失败日志、死信队列和监控告警；
10. 压测订单创建、消息消费和补偿扫描。

## 总结

订单超时取消的重点不是选择某个延迟工具，而是把状态流转、幂等、补偿和监控做好。

中小规模系统可以先用定时任务实现，配合合理索引和批量扫描，成本最低。订单量提升后，建议使用 MQ 延迟消息承担主要触发，再用定时任务兜底。Redis 过期监听适合辅助通知，不建议承担核心链路。时间轮性能很好，但需要解决持久化和集群部署问题。

最终推荐方案是：MQ 延迟消息 + 定时任务补偿 + 数据库条件更新 + 业务幂等。这样既能保证较好的实时性，也能保证异常情况下的最终一致性。
