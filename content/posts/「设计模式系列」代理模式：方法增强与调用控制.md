## 模式定位

代理模式的核心是：不直接访问目标对象，而是通过代理对象间接访问。

代理对象和目标对象通常实现相同接口，对调用方来说，调用代理对象和调用真实对象的方式保持一致。代理对象可以在方法调用前后增加额外逻辑，比如日志、权限校验、事务控制、缓存处理等。

可以用一句话理解：

```text
调用方 -> 代理对象 -> 目标对象
```

代理模式常见于这些场景：

- 方法执行前后记录日志；
- 统一权限校验；
- 事务开启、提交、回滚；
- 缓存查询和缓存更新；
- RPC 远程服务调用；
- 延迟加载；
- 接口限流和监控埋点。

Spring AOP、MyBatis Mapper、Dubbo 服务调用、Feign 接口调用背后都能看到代理思想。

## 静态代理

先看一个订单查询接口：

```java
public interface OrderService {

    OrderDTO queryOrder(Long orderId);
}
```

真实业务实现只负责查询订单：

```java
public class OrderServiceImpl implements OrderService {

    @Override
    public OrderDTO queryOrder(Long orderId) {
        // 真实业务逻辑：查询数据库或调用订单中心
        System.out.println("查询订单，orderId = " + orderId);
        return new OrderDTO(orderId, "PAID");
    }
}
```

如果现在要在查询前后增加日志，最直接的方式是在业务方法里写日志。但这样会让业务代码和非业务逻辑耦合。

可以增加一个代理类：

```java
public class OrderServiceProxy implements OrderService {

    private final OrderService target;

    public OrderServiceProxy(OrderService target) {
        // 代理对象持有真实目标对象
        this.target = target;
    }

    @Override
    public OrderDTO queryOrder(Long orderId) {
        long start = System.currentTimeMillis();
        System.out.println("开始查询订单，orderId = " + orderId);

        try {
            // 真正的业务逻辑仍然交给目标对象执行
            return target.queryOrder(orderId);
        } finally {
            long cost = System.currentTimeMillis() - start;
            System.out.println("订单查询结束，耗时 " + cost + " ms");
        }
    }
}
```

调用方使用代理对象：

```java
OrderService target = new OrderServiceImpl();
OrderService proxy = new OrderServiceProxy(target);

OrderDTO order = proxy.queryOrder(1001L);
```

静态代理的优点是简单直观，缺点也很明显：

- 每个接口都要手写代理类；
- 方法多时重复代码很多；
- 新增接口或方法时代理类也要改；
- 不适合做通用增强。

如果只是给少量类做固定增强，静态代理可以接受。但在框架层面，它的扩展性不够。

## JDK 动态代理

JDK 动态代理可以在运行时生成代理对象，不需要手写具体代理类。

它有一个前提：目标类必须实现接口。

继续使用 `OrderService` 接口，可以写一个通用调用处理器：

```java
import java.lang.reflect.InvocationHandler;
import java.lang.reflect.Method;

public class LogInvocationHandler implements InvocationHandler {

    private final Object target;

    public LogInvocationHandler(Object target) {
        // target 是真实业务对象，代理最终会把调用转发给它
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        long start = System.currentTimeMillis();
        System.out.println("调用方法：" + method.getName());

        try {
            // 通过反射调用目标对象上的真实方法
            return method.invoke(target, args);
        } finally {
            long cost = System.currentTimeMillis() - start;
            System.out.println("方法执行耗时：" + cost + " ms");
        }
    }
}
```

创建代理对象：

```java
import java.lang.reflect.Proxy;

public class JdkProxyDemo {

    public static void main(String[] args) {
        OrderService target = new OrderServiceImpl();

        OrderService proxy = (OrderService) Proxy.newProxyInstance(
                target.getClass().getClassLoader(),
                target.getClass().getInterfaces(),
                new LogInvocationHandler(target)
        );

        // 调用代理对象的方法，最终会进入 InvocationHandler#invoke
        proxy.queryOrder(1001L);
    }
}
```

执行链路可以理解成：

```text
proxy.queryOrder()
-> LogInvocationHandler.invoke()
-> method.invoke(target, args)
-> OrderServiceImpl.queryOrder()
```

JDK 动态代理适合给一批接口统一增加增强逻辑。比如日志、耗时统计、权限校验，都可以写在 `InvocationHandler` 里。

它的限制是目标类必须实现接口。如果目标类没有接口，就需要考虑 CGLIB。

## CGLIB 代理

CGLIB 的思路是：运行时生成目标类的子类，通过重写方法完成增强。

它不要求目标类实现接口，但目标类不能是 `final`，被代理的方法也不能是 `final`。

示例目标类：

```java
public class UserService {

    public UserDTO queryUser(Long userId) {
        // 真实业务逻辑：查询用户信息
        System.out.println("查询用户，userId = " + userId);
        return new UserDTO(userId, "Shaun");
    }
}
```

使用 CGLIB 创建代理对象：

```java
import org.springframework.cglib.proxy.Enhancer;
import org.springframework.cglib.proxy.MethodInterceptor;
import org.springframework.cglib.proxy.MethodProxy;

import java.lang.reflect.Method;

public class CglibProxyFactory implements MethodInterceptor {

    public Object createProxy(Class<?> targetClass) {
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(targetClass);

        // 当前对象负责拦截代理方法调用
        enhancer.setCallback(this);
        return enhancer.create();
    }

    @Override
    public Object intercept(Object obj, Method method, Object[] args, MethodProxy proxy) throws Throwable {
        long start = System.currentTimeMillis();
        System.out.println("CGLIB 调用方法：" + method.getName());

        try {
            // 调用父类方法，也就是目标类中的真实业务逻辑
            return proxy.invokeSuper(obj, args);
        } finally {
            long cost = System.currentTimeMillis() - start;
            System.out.println("CGLIB 方法耗时：" + cost + " ms");
        }
    }
}
```

调用方式：

```java
CglibProxyFactory factory = new CglibProxyFactory();
UserService userService = (UserService) factory.createProxy(UserService.class);

userService.queryUser(1001L);
```

CGLIB 和 JDK 动态代理的区别可以简单理解为：

| 方式 | 原理 | 要求 | 常见场景 |
| --- | --- | --- | --- |
| JDK 动态代理 | 基于接口生成代理类 | 目标类实现接口 | 接口驱动的服务增强 |
| CGLIB 代理 | 基于继承生成子类 | 类和方法不能是 final | 没有接口的类增强 |

## Spring AOP

Spring AOP 底层就使用了代理机制。

当我们写一个切面：

```java
@Aspect
@Component
public class ApiLogAspect {

    @Around("execution(* com.example.demo.service..*(..))")
    public Object around(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        String methodName = joinPoint.getSignature().toShortString();

        System.out.println("开始执行：" + methodName);

        try {
            // proceed 会继续调用目标方法
            return joinPoint.proceed();
        } finally {
            long cost = System.currentTimeMillis() - start;
            System.out.println("执行结束：" + methodName + "，耗时 " + cost + " ms");
        }
    }
}
```

业务类本身不需要写日志逻辑：

```java
@Service
public class OrderAppService {

    public OrderDTO queryOrder(Long orderId) {
        // 这里保持纯业务逻辑，日志由 AOP 统一处理
        return new OrderDTO(orderId, "PAID");
    }
}
```

Spring 会为符合切点规则的 Bean 创建代理对象。调用方注入的通常不是原始对象，而是代理对象。

调用链路类似：

```text
Controller 调用 OrderAppService 代理对象
-> 进入 ApiLogAspect.around()
-> joinPoint.proceed()
-> 执行 OrderAppService.queryOrder()
-> 返回结果并打印耗时
```

这就是代理模式在 Spring 项目里最常见的应用。

## 常见场景

### 事务控制

`@Transactional` 就是代理模式的典型应用。

```java
@Transactional(rollbackFor = Exception.class)
public void createOrder(CreateOrderRequest request) {
    // 代理对象在方法执行前开启事务
    orderRepository.save(request);

    // 方法正常结束时提交事务，抛出异常时回滚事务
    stockRepository.deduct(request.getSkuId(), request.getCount());
}
```

事务逻辑没有写在业务方法里，而是由 Spring 代理对象在方法执行前后统一处理。

### 权限校验

后台管理系统里，某些接口需要统一校验操作权限。

可以用注解配合 AOP：

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequirePermission {

    String value();
}
```

```java
@RequirePermission("order:refund")
public void refund(Long orderId) {
    // 真正的退款业务逻辑
    refundService.refund(orderId);
}
```

代理对象在执行 `refund` 前读取注解，检查当前用户是否拥有 `order:refund` 权限。

### 缓存增强

缓存也很适合用代理做统一增强：

```java
@Cacheable(cacheNames = "user", key = "#userId")
public UserDTO queryUser(Long userId) {
    // 缓存未命中时才会执行真实查询逻辑
    return userRepository.findById(userId);
}
```

调用方感知不到缓存逻辑，只关心方法返回值。

## 常见误区

### 内部方法调用不生效

Spring AOP 是基于代理对象实现的。一个类内部自己调用自己的方法时，不会经过代理对象。

```java
@Service
public class OrderService {

    public void createOrder() {
        // 这是 this.pay()，不会经过 Spring 代理对象
        pay();
    }

    @Transactional(rollbackFor = Exception.class)
    public void pay() {
        // 事务可能不会按预期生效
    }
}
```

调用链路是：

```text
外部调用代理对象 createOrder()
-> 进入目标对象 createOrder()
-> 目标对象内部 this.pay()
-> 没有经过代理对象
```

解决方式通常有几种：

- 把 `pay()` 拆到另一个 Spring Bean；
- 通过代理对象调用，而不是 `this` 调用；
- 调整事务边界，把注解放在外部入口方法。

### final 方法无法被 CGLIB 增强

CGLIB 依赖继承和方法重写。如果类或方法被 `final` 修饰，就无法正常生成可增强的子类方法。

```java
public final class UserService {

    public final UserDTO queryUser(Long userId) {
        return new UserDTO(userId, "Shaun");
    }
}
```

这种写法不适合 CGLIB 代理。

### 代理对象类型判断错误

有些代码会用具体实现类接收 Bean：

```java
@Autowired
private OrderServiceImpl orderService;
```

如果 Spring 创建的是 JDK 动态代理，代理对象实现了接口，但不一定是 `OrderServiceImpl` 类型，可能导致注入或类型转换问题。

更推荐面向接口注入：

```java
@Autowired
private OrderService orderService;
```

这也符合面向接口编程的习惯。

## 选型建议

在普通后端项目里，可以这样判断：

| 场景 | 建议 |
| --- | --- |
| 少量固定增强 | 静态代理 |
| 目标类有接口 | JDK 动态代理 |
| 目标类没有接口 | CGLIB 代理 |
| Spring Boot 业务增强 | Spring AOP |
| RPC 接口调用 | 动态代理思想 |
| 方法内部调用增强 | 注意代理失效问题 |

如果在 Spring 项目中做日志、权限、事务、缓存、监控，优先考虑 AOP，不建议手写大量代理类。

如果要做框架或中间件，理解 JDK 动态代理和 CGLIB 会更重要，因为很多“看起来像本地方法调用”的能力，本质上都是代理对象在背后做了拦截和转发。

## 总结

代理模式的价值在于控制访问和增强能力。调用方看似调用普通对象，实际调用的是代理对象，代理对象可以在转发调用前后增加统一逻辑。

静态代理适合理解基础思想，JDK 动态代理适合接口增强，CGLIB 适合类增强，Spring AOP 则是代理模式在业务项目中最常见的落地方式。

掌握代理模式后，再看事务、缓存、权限、RPC、Mapper 接口这些机制，会更容易理解它们为什么可以“不改业务代码，却能增加额外能力”。
