## 面试官想问什么？

面试里被问到“你们的后端接口是怎么设计的”，不要只回答“用 RESTful，返回 JSON”。这个问题通常想考察三个方面：

- 有没有统一的接口规范；
- 能不能考虑业务边界和异常场景；
- 是否具备工程化意识，比如幂等、鉴权、版本兼容、监控和文档。

一个比较完整的回答可以按这个顺序展开：

```text
业务场景确认
-> 资源和 URL 设计
-> 请求参数设计
-> 响应结构设计
-> 参数校验和错误码
-> 鉴权、幂等、安全
-> 性能和分页
-> 版本兼容
-> 文档、联调、监控
```

如果能结合具体项目接口，比如订单创建、用户登录、文件上传、审批提交，回答会更真实。

## 设计前先定边界

接口设计前，不应该直接打开 IDEA 写 Controller。先把几个问题确认清楚：

| 问题 | 说明 |
| --- | --- |
| 谁来调用 | 前端、App、第三方系统、内部服务 |
| 做什么业务 | 查询、创建、提交、审核、取消、回调 |
| 是否写操作 | 写操作要考虑事务、幂等和权限 |
| 是否高频访问 | 高频接口要考虑缓存、限流、分页 |
| 是否对外开放 | 开放接口要考虑签名、时间戳、防重放 |
| 是否需要兼容 | App、小程序、第三方调用方升级慢，要考虑版本 |
| 是否涉及敏感数据 | 身份证、手机号、金额、权限数据要脱敏和审计 |

比如一个“创建订单接口”，不能只看成保存订单，还要确认：

- 是否需要锁定库存；
- 是否需要使用优惠券；
- 是否需要校验商品上下架；
- 是否允许重复提交；
- 创建失败后库存如何回滚；
- 支付超时后如何取消订单；
- 接口是否会被用户连续点击。

接口的复杂度往往不在入参和出参，而在背后的业务一致性。

## URL 设计

内部业务接口一般会采用 REST 风格。核心原则是：URL 表达资源，HTTP 方法表达动作。

常见设计：

| 场景 | 方法 | URL |
| --- | --- | --- |
| 查询订单列表 | `GET` | `/api/v1/orders` |
| 查询订单详情 | `GET` | `/api/v1/orders/{orderId}` |
| 创建订单 | `POST` | `/api/v1/orders` |
| 修改订单备注 | `PATCH` | `/api/v1/orders/{orderId}/remark` |
| 取消订单 | `POST` | `/api/v1/orders/{orderId}/cancel` |
| 删除草稿 | `DELETE` | `/api/v1/order-drafts/{draftId}` |

实际业务里不必机械追求所有接口都完全 RESTful。比如“取消订单”“提交审批”“确认收货”这类动作型接口，用 `/cancel`、`/submit`、`/confirm` 反而更清晰。

接口命名要避免几个问题：

```text
/getOrderInfo
/createNewOrder
/updateOrderStatusByUser
/doCancel
/queryData
```

这些 URL 把动作和实现细节揉在一起，后期会越来越乱。

更推荐：

```text
GET  /api/v1/orders/{orderId}
POST /api/v1/orders
POST /api/v1/orders/{orderId}/cancel
GET  /api/v1/users/{userId}/orders
```

版本可以放在路径里，例如 `/api/v1`。对内部系统来说，路径版本最直观；对开放平台来说，也可以通过 Header 传版本。

## 请求参数

请求参数一般分四类：

| 类型 | 适合放什么 |
| --- | --- |
| Path Variable | 资源 ID，例如 `orderId`、`userId` |
| Query Parameter | 查询条件、分页、排序 |
| Request Body | 创建、修改类接口的复杂对象 |
| Header | Token、TraceId、客户端版本、幂等 Key |

订单列表查询可以这样设计：

```http
GET /api/v1/orders?status=UNPAID&pageNo=1&pageSize=20&sort=createdAt,desc
Authorization: Bearer xxx
X-Request-Id: 7f8f0c9d4c5c4d71
```

创建订单适合使用 JSON Body：

```json
{
  "addressId": 1001,
  "couponId": 2001,
  "items": [
    {
      "skuId": 3001,
      "quantity": 2
    }
  ],
  "remark": "工作日配送"
}
```

几个实践建议：

- 金额不要用 `double`，优先使用分为单位的 `Long` 或 `BigDecimal`；
- 时间字段格式统一，建议明确时区；
- 枚举值要有明确范围，例如 `UNPAID`、`PAID`、`CANCELED`；
- 分页参数要限制最大 `pageSize`；
- 查询接口不要把大量复杂条件都塞进路径；
- 写接口不要依赖前端传用户 ID，用户身份应从登录态解析。

## 响应结构

响应结构要统一，否则前端每接一个接口都要单独适配。

一种常见结构：

```json
{
  "code": "0",
  "message": "success",
  "data": {
    "orderId": 1001,
    "orderNo": "ORD202607050001",
    "status": "UNPAID"
  },
  "traceId": "7f8f0c9d4c5c4d71"
}
```

分页结构也要固定：

```json
{
  "code": "0",
  "message": "success",
  "data": {
    "pageNo": 1,
    "pageSize": 20,
    "total": 245,
    "records": [
      {
        "orderId": 1001,
        "orderNo": "ORD202607050001",
        "status": "UNPAID"
      }
    ]
  },
  "traceId": "7f8f0c9d4c5c4d71"
}
```

是否使用 HTTP 状态码，需要团队统一。

比较常见的做法是：

| HTTP 状态码 | 场景 |
| --- | --- |
| `200` | 业务正常返回，包括业务失败结果 |
| `400` | 参数格式错误、字段类型错误 |
| `401` | 未登录或 Token 无效 |
| `403` | 无权限 |
| `404` | 资源不存在 |
| `429` | 请求过于频繁 |
| `500` | 服务端异常 |

业务错误码放在响应体里，例如：

```json
{
  "code": "ORDER_STOCK_NOT_ENOUGH",
  "message": "商品库存不足",
  "data": null,
  "traceId": "7f8f0c9d4c5c4d71"
}
```

错误码不要随便写成 `-1`、`9999`。错误码应该能看出业务含义，也方便前端和测试定位。

## 参数校验

参数校验应该靠统一机制完成，不要在业务代码里堆大量 `if`。

Spring Boot 项目里可以使用 Bean Validation，Jakarta Bean Validation 规范文档可以查看：[Jakarta Bean Validation](https://beanvalidation.org/)。

创建订单请求对象：

```java
public class CreateOrderRequest {

    @NotNull(message = "收货地址不能为空")
    private Long addressId;

    private Long couponId;

    @NotEmpty(message = "订单商品不能为空")
    @Valid
    private List<CreateOrderItemRequest> items;

    @Size(max = 200, message = "备注最多 200 个字符")
    private String remark;

    // getter/setter 省略
}
```

订单商品请求对象：

```java
public class CreateOrderItemRequest {

    @NotNull(message = "SKU 不能为空")
    private Long skuId;

    @NotNull(message = "购买数量不能为空")
    @Min(value = 1, message = "购买数量必须大于 0")
    private Integer quantity;

    // getter/setter 省略
}
```

Controller 里只需要加 `@Valid`：

```java
@PostMapping("/api/v1/orders")
public ApiResponse<CreateOrderResponse> createOrder(@Valid @RequestBody CreateOrderRequest request) {
    // 参数基础校验已经在进入业务逻辑前完成
    CreateOrderResponse response = orderAppService.createOrder(request);
    return ApiResponse.success(response);
}
```

全局异常处理负责把校验错误转换成统一响应：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ApiResponse<Void> handleValidException(MethodArgumentNotValidException ex) {
        String message = ex.getBindingResult()
                .getFieldErrors()
                .stream()
                .findFirst()
                .map(FieldError::getDefaultMessage)
                .orElse("参数校验失败");

        // 对外返回统一错误码，前端可以按 code 做提示或特殊处理
        return ApiResponse.fail("PARAM_INVALID", message);
    }

    @ExceptionHandler(BusinessException.class)
    public ApiResponse<Void> handleBusinessException(BusinessException ex) {
        // 业务异常只暴露可理解的信息，不直接返回堆栈
        return ApiResponse.fail(ex.getCode(), ex.getMessage());
    }
}
```

这样业务代码可以更聚焦业务规则，例如商品是否可售、库存是否足够、优惠券是否可用。

## 幂等设计

只要是写操作，就要考虑幂等。

常见需要幂等的接口：

- 创建订单；
- 提交支付；
- 取消订单；
- 退款申请；
- 审批提交；
- 回调通知；
- 重试任务。

前端按钮禁用只能减少重复点击，不能替代后端幂等。网络重试、用户刷新、App 重放、MQ 重复消费都会导致同一个请求被多次处理。

创建订单可以使用 `Idempotency-Key`：

```http
POST /api/v1/orders
Authorization: Bearer xxx
Idempotency-Key: 6a9f77b7c7b24a0b9f5d2c8e6d91a001
```

后端可以把用户 ID + 幂等 Key 作为唯一约束：

```sql
CREATE TABLE `idempotent_record` (
  `id` BIGINT NOT NULL PRIMARY KEY,
  `user_id` BIGINT NOT NULL,
  `idempotent_key` VARCHAR(128) NOT NULL,
  `business_type` VARCHAR(64) NOT NULL,
  `business_id` BIGINT DEFAULT NULL,
  `status` VARCHAR(32) NOT NULL,
  `created_at` DATETIME NOT NULL,
  `updated_at` DATETIME NOT NULL,
  UNIQUE KEY `uk_user_key_type` (`user_id`, `idempotent_key`, `business_type`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

取消订单可以通过状态条件更新保证幂等：

```sql
UPDATE orders
SET status = 'CANCELED',
    cancel_time = NOW(),
    updated_at = NOW()
WHERE id = #{orderId}
  AND user_id = #{userId}
  AND status = 'UNPAID';
```

这条 SQL 重复执行时，只有第一次能更新成功。后续执行返回 `0`，服务端再查询当前订单状态，给调用方返回合理结果。

## 鉴权与安全

接口设计里，鉴权不能后补。不同类型接口要有不同安全策略。

内部后台接口通常需要：

- 登录校验；
- 角色权限；
- 菜单权限；
- 按钮权限；
- 数据权限。

开放接口通常需要：

- `appId`；
- `timestamp`；
- `nonce`；
- `signature`；
- 防重放校验；
- IP 白名单；
- 访问频率限制。

不要把敏感参数放在 URL 里：

```text
/api/v1/users?phone=13800138000&idCard=xxx
```

URL 可能被网关日志、浏览器历史、监控系统记录。敏感信息应该尽量放在 Body 里，并做好脱敏日志。

返回数据也要控制字段。普通用户查询订单，不应该返回内部成本价、供应商 ID、风控标记等内部字段。

## 性能细节

接口设计阶段就要考虑性能边界，不要等线上慢了再补救。

### 分页

列表接口必须分页，并限制最大页大小：

```java
public class PageRequest {

    @Min(value = 1, message = "页码必须大于 0")
    private Integer pageNo = 1;

    @Min(value = 1, message = "每页数量必须大于 0")
    @Max(value = 100, message = "每页最多查询 100 条")
    private Integer pageSize = 20;

    // 统一计算 offset，避免各个 Mapper 重复写分页逻辑
    public int offset() {
        return (pageNo - 1) * pageSize;
    }
}
```

如果数据量很大，深分页要改成游标分页：

```http
GET /api/v1/orders?lastId=100000&pageSize=20
```

对应 SQL：

```sql
SELECT id, order_no, status, created_at
FROM orders
WHERE user_id = #{userId}
  AND id < #{lastId}
ORDER BY id DESC
LIMIT #{pageSize};
```

### 字段控制

列表页不要返回详情页字段。比如订单列表只需要订单号、状态、金额、创建时间，不需要返回商品详情、物流轨迹、发票信息。

可以拆成两个接口：

```text
GET /api/v1/orders              订单列表
GET /api/v1/orders/{orderId}    订单详情
```

### 超时和降级

如果一个接口依赖多个外部服务，要设置超时时间和降级策略。

```text
订单详情接口
-> 查询订单库
-> 查询商品服务
-> 查询物流服务
-> 查询优惠券服务
```

物流服务超时不应该直接拖垮订单详情接口，可以返回基础订单信息，并把物流状态标记为“暂时不可用”。

## 版本兼容

接口上线后，最怕随意改字段。

几个基本原则：

- 新增字段通常兼容；
- 删除字段风险很高；
- 修改字段含义风险很高；
- 枚举值新增要提前通知调用方；
- 金额、时间、状态字段不能随意改格式；
- App、小程序、第三方系统升级慢，要保留旧版本一段时间。

比如 `status` 原来只有：

```text
UNPAID
PAID
CANCELED
```

后面新增：

```text
REFUNDING
REFUNDED
```

前端如果没有默认兜底，就可能展示异常。所以枚举字段要提前约定未知值处理方式。

## 文档与联调

接口文档要和代码一起维护，不能只靠口头约定。

常见做法是使用 OpenAPI 规范，OpenAPI 官方规范可以查看：[OpenAPI Specification](https://spec.openapis.org/oas/latest.html)。如果团队使用 Swagger UI，可以参考：[Swagger UI](https://swagger.io/tools/swagger-ui/)。

一份接口文档至少要包含：

- 接口名称；
- URL 和 HTTP 方法；
- 请求 Header；
- 请求参数说明；
- 响应字段说明；
- 错误码；
- 权限要求；
- 示例请求；
- 示例响应；
- 变更记录。

接口评审时，后端、前端、测试最好一起确认字段含义，尤其是状态枚举、金额单位、时间格式、空值规则。

空值规则要提前约定：

```text
列表无数据：返回 []，不要返回 null
对象不存在：返回明确错误码或 null，团队统一
字符串无内容：返回 "" 还是 null，团队统一
数字无值：返回 null 还是 0，团队统一
```

这些细节看起来小，联调时最容易扯皮。

## 一个完整示例

以创建订单接口为例，可以这样定义。

### 接口说明

```text
接口名称：创建订单
HTTP 方法：POST
接口地址：/api/v1/orders
权限要求：用户已登录
幂等要求：需要 Idempotency-Key
Content-Type：application/json
```

### 请求 Header

```http
Authorization: Bearer {token}
Idempotency-Key: {uuid}
X-Request-Id: {traceId}
```

### 请求 Body

```json
{
  "addressId": 1001,
  "couponId": 2001,
  "items": [
    {
      "skuId": 3001,
      "quantity": 2
    }
  ],
  "remark": "工作日配送"
}
```

### 成功响应

```json
{
  "code": "0",
  "message": "success",
  "data": {
    "orderId": 9001,
    "orderNo": "ORD202607050001",
    "status": "UNPAID",
    "payExpireTime": "2026-07-05 23:50:00"
  },
  "traceId": "7f8f0c9d4c5c4d71"
}
```

### 常见错误码

| 错误码 | 含义 |
| --- | --- |
| `PARAM_INVALID` | 参数校验失败 |
| `ADDRESS_NOT_FOUND` | 收货地址不存在 |
| `SKU_NOT_SALEABLE` | 商品不可售 |
| `STOCK_NOT_ENOUGH` | 库存不足 |
| `COUPON_NOT_AVAILABLE` | 优惠券不可用 |
| `IDEMPOTENT_KEY_REQUIRED` | 缺少幂等 Key |
| `ORDER_CREATE_DUPLICATED` | 重复创建订单 |

### Controller 示例

```java
@RestController
@RequestMapping("/api/v1/orders")
public class OrderController {

    private final OrderAppService orderAppService;

    public OrderController(OrderAppService orderAppService) {
        this.orderAppService = orderAppService;
    }

    @PostMapping
    public ApiResponse<CreateOrderResponse> createOrder(
            @RequestHeader("Idempotency-Key") String idempotencyKey,
            @Valid @RequestBody CreateOrderRequest request,
            LoginUser loginUser) {

        CreateOrderCommand command = new CreateOrderCommand();
        command.setUserId(loginUser.getUserId());
        command.setIdempotencyKey(idempotencyKey);
        command.setAddressId(request.getAddressId());
        command.setCouponId(request.getCouponId());
        command.setItems(request.getItems());
        command.setRemark(request.getRemark());

        // Controller 只做参数接收和对象转换，不写复杂业务逻辑
        CreateOrderResponse response = orderAppService.createOrder(command);
        return ApiResponse.success(response);
    }
}
```

这个接口背后的核心处理链路可以是：

```text
登录校验
-> 参数校验
-> 幂等 Key 校验
-> 地址校验
-> 商品校验
-> 库存锁定
-> 优惠券锁定
-> 创建订单
-> 发送订单超时取消延迟消息
-> 返回订单号和支付过期时间
```

这种回答会比“我们用 POST 创建订单，GET 查询订单”更有说服力。

## 面试回答模板

面试时可以这样组织回答：

```text
我们项目里的接口设计会先确认业务边界，比如调用方、是否写操作、是否需要幂等、是否涉及权限和敏感数据。

接口风格上主要按 REST 思路设计，URL 表示资源，HTTP 方法表示动作。查询类接口用 GET，创建类接口用 POST，局部修改用 PATCH。像取消订单、提交审核这种业务动作，会使用 /cancel、/submit 这类语义明确的路径。

请求参数会区分 Path、Query、Body 和 Header。资源 ID 放 Path，查询条件和分页放 Query，复杂写入对象放 Body，Token、TraceId、幂等 Key 放 Header。

响应结构是统一的，包含 code、message、data、traceId。业务错误通过错误码表达，参数校验和业务异常由全局异常处理统一返回。

写接口会重点考虑幂等，比如创建订单使用 Idempotency-Key，取消订单用状态条件更新。权限方面会区分登录校验、角色权限和数据权限，敏感字段会做脱敏。

另外会考虑分页、限流、超时、版本兼容和接口文档。上线后通过日志、traceId、接口耗时、错误码统计来排查问题。
```

如果面试官继续追问，可以结合项目里的一个接口深入讲，比如订单创建、订单取消、文件上传、支付回调，这样更容易体现真实经验。

## 总结

后端接口设计不是单纯定义 URL 和 JSON 字段，而是把业务规则、调用方体验、系统安全、数据一致性和后期维护都考虑进去。

一个合格的接口，应该具备清晰的资源命名、稳定的请求响应结构、统一的错误码、可靠的参数校验、必要的幂等控制、明确的权限边界、可追踪的日志和可维护的文档。
